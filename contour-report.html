Dimension details
1 Integration Surface OTLP gRPC for traces; Prometheus-only for metrics; no log export ▶

What we observed

Contour uses Envoy as its data plane and routes trace export through a repurposed ExtensionService CRD (normally for external authorization) pointing at the OTel Collector gRPC endpoint. Traces flow via OTLP gRPC, confirmed by 9 trace batches in the evaluation cluster. Metrics are Prometheus-only — both Contour (:8000/metrics) and Envoy (:8002/stats/prometheus) expose scrape endpoints with no OTLP metrics path. Logs remain on stdout with no export mechanism. The two working signals use entirely different collection models (OTLP push for traces, Prometheus pull for metrics), requiring separate configuration paths.

What's working well

  • OTLP gRPC trace export is supported and documented — users can connect to an existing OTel Collector without sidecars using the ExtensionService CRD and tracing.extensionService configmap field.
  • Rich Prometheus metrics are available from both Contour (14 control-plane metrics including contour_build_info) and Envoy (469 data-plane metrics), providing comprehensive operational visibility for users already running Prometheus.

Opportunities

Unify the metrics integration surface — Adding an OTLP metrics export path (or explicitly documenting the Prometheus receiver pipeline as the supported integration contract) would bring metrics in line with the trace export model and reduce the per-signal configuration burden for operators.
Enable structured log export — Envoy access logs could be configured in JSON format and collected via the OTel Collector filelog receiver, enabling log-trace correlation. Contour's control-plane logs could similarly be structured for collection via OTLP.

Checklist

  • ✓ OTLP is supported for traces — users can connect to an existing OTel Collector without sidecars
  • ✓ Tracing is documented with a dedicated guide and design document (PR #5043)
  • ✗ No OTLP metrics export — Prometheus is the only metrics path; no unified integration surface
  • ✗ No OTLP log export; no standard OTEL_* environment variable support for configuration
0 Semantic Conventions Deprecated HTTP attributes (http.method, http.url, http.status_code); non-standard custom keys ▶

What we observed

Envoy emits trace attributes using the older OTel HTTP semantic conventions throughout. The ingress span carries http.method, http.url, http.status_code (as a string), and http.protocol — all deprecated in favor of http.request.method, url.full, http.response.status_code (integer), and network.protocol.version. Additional non-standard attributes include upstream_cluster, upstream_cluster.name, peer.address, podName (camelCase), and podNamespace. Notably, http.route is absent — the single most operationally useful attribute for an ingress controller. Metric names use Envoy/Prometheus conventions (envoy_*, contour_*) rather than OTel semantic conventions.

What's working well

  • HTTP semantics are present in some form — status codes, methods, URLs, and protocol versions are all captured, providing a foundation that can be upgraded to current conventions as Envoy's OTel tracer evolves.
  • The contour_build_info metric and Envoy telemetry SDK attributes (telemetry.sdk.name=envoy, telemetry.sdk.version) provide useful identity context for tooling that understands these conventions.

Opportunities

Adopt current OTel HTTP semantic conventions — Replacing http.method → http.request.method, http.url → url.full, http.status_code (string) → http.response.status_code (int), and http.protocol → network.protocol.version would align with stable OTel HTTP semconv. Adding http.route would be the highest-value single change for an ingress controller. This is primarily an Envoy upstream change, but Contour can advocate for it.
Normalize custom attribute naming to OTel dot-notation — Attributes like podName and podNamespace (camelCase) should become k8s.pod.name and k8s.namespace.name to follow OTel resource attribute conventions. user_agent should become user_agent.original. These changes make telemetry interpretable without Envoy-specific knowledge.
1 Resource Attributes & Configuration service.name present but no service.version; no OTEL_* env var support; identity inconsistent across signals ▶

What we observed

Contour/Envoy natively emits service.name (configured as "contour" via the Contour configmap), telemetry.sdk.language=cpp, telemetry.sdk.name=envoy, and telemetry.sdk.version (a commit hash + version string). However, service.version, service.instance.id, and service.namespace are all absent. Contour already knows its version — it appears in the contour_build_info metric as version=v1.32.1 — but this is not propagated to trace resource attributes. Neither Contour nor Envoy support standard OTEL_* environment variables for configuration; all telemetry setup goes through the Contour configmap or ContourConfiguration CRD. The service.name=contour in traces actually refers to Envoy's activity (not the control plane), creating a conceptual mismatch.

What's working well

  • service.name is present and stable on all trace exports, providing the minimum identity anchor for trace-based queries and dashboards.
  • Kubernetes metadata (k8s.pod.name, k8s.daemonset.name, k8s.namespace.name, k8s.node.name) is enriched by the OTel Collector's k8sattributes processor — an acceptable and recommended pattern for Kubernetes-native projects.

Opportunities

Emit service.version and service.instance.id — Contour already knows its version (visible in contour_build_info with label version=v1.32.1). Propagating this as service.version in trace resource attributes would enable version-based filtering. Setting service.instance.id (using pod UID) would distinguish individual Envoy pods without relying on collector-added k8s.pod.name.
Clarify service identity between control plane and data plane — Currently service.name=contour in traces refers to Envoy's data-plane activity, while the Contour control plane emits no OTLP traces at all. Using distinct service names (e.g., contour-envoy for data plane, contour for control plane) would eliminate the conceptual mismatch and make the trace topology self-explanatory.
2 Trace Modeling & Context Propagation Coherent ingress→egress→backend trace tree; W3C Trace Context propagated end-to-end ▶

What we observed

Contour's trace modeling is the standout strength of its OTel implementation. A complete HTTP request produces a coherent three-level parent-child tree: ingress (SERVER) → router demo_otel-eval-backend_3000 egress (CLIENT) → GET / (SERVER, backend). All 20 Envoy traces in the evaluation shared trace IDs with corresponding backend spans — 100% cross-service correlation. W3C Trace Context (traceparent header) is propagated end-to-end: Envoy reads incoming traceparent headers and injects them into upstream requests, verified with trace ID 4bf92f3577b34da6a3ce929d0e0e4736. Span kinds are semantically correct: SERVER at ingress, CLIENT at egress.

What's working well

  • W3C Trace Context propagation works end-to-end — 100% of traffic-generated traces formed complete cross-service parent-child chains across the ingress boundary, verified across 20 Envoy spans and 244 backend spans.
  • Span kinds are semantically correct — kind=SERVER for inbound ingress spans and kind=CLIENT for upstream egress spans — enabling correct trace visualization and RED metric computation in observability backends.

Opportunities

Improve span naming for user-facing operations — The current span names ingress and router demo_otel-eval-backend_3000 egress reflect Envoy's internal component architecture. Adopting HTTP semantic naming (e.g., GET /path or HTTP GET) for the ingress span would make traces immediately interpretable without Envoy knowledge, and align with how backend services name their spans.
Verify trace topology for error paths and retries — The evaluation confirmed correct trace topology for the happy path. Extending verification to error responses (5xx), Envoy retries, TLS termination, and upstream timeouts would confirm that the trace model holds under all conditions and that response_flags (e.g., UH for upstream unhealthy) are correctly surfaced.
1 Multi-Signal Observability Traces (OTLP) and metrics (Prometheus) both flow; no structured logs; no cross-signal correlation ▶

What we observed

Two of three signals are flowing: traces via OTLP gRPC (9 batches confirmed) and metrics via Prometheus scrape (100 batches confirmed). Logs are absent — the log output file was confirmed empty (0 bytes). The two working signals use incompatible collection models: OTLP push for traces vs. Prometheus pull for metrics. There is no cross-signal correlation — traces and metrics share service.name=contour in name but use different attribute schemas (OTel span attributes vs. Envoy/Prometheus metric labels). No trace_id appears on metrics, and logs cannot be correlated because they are not exported. The contour_build_info metric carries version=v1.32.1 as a label, but this version is not present in trace resource attributes as service.version.

What's working well

  • Both traces and metrics are documented and production-ready — operators can set up end-to-end trace collection and Prometheus scraping independently, with official guides for both.
  • The Envoy metrics set is comprehensive (469 envoy_* metrics) covering request rates, error rates, connection counts, circuit breaker state, and more — giving operators deep data-plane visibility through Prometheus.

Opportunities

Add structured log export for log-trace correlation — Enabling JSON-formatted Envoy access logs collected via the OTel Collector filelog receiver would add the third signal and enable log-trace correlation using the x-request-id / guid:x-request-id already present in spans. This is the highest-impact multi-signal improvement available.
Bridge trace and metric identity with shared service.version — The contour_build_info metric already carries version=v1.32.1. Surfacing this same version as service.version in trace resource attributes would allow operators to correlate a spike in error-rate metrics with a specific deployment version without manual cross-referencing.
1 Audience & Signal Quality Span names reflect Envoy internals; useful operational attributes present; some noise ▶

What we observed

Envoy emits two span types per request: ingress (all inbound requests, regardless of method, path, or route) and router demo_otel-eval-backend_3000 egress (upstream call, encoding the internal cluster name). Neither name is self-explanatory to an OTel user unfamiliar with Envoy internals. The ingress span carries 18 attributes — some useful (upstream_cluster.name, http.status_code, upstream_address, response_flags, podName), some low-value (zone: "", downstream_cluster: "-", node_id duplicating podName). The metric set has 554 unique names (14 contour_*, 469 envoy_*), comprehensive but noisy — most operators need only a small subset for SLO dashboards.

What's working well

  • Useful operational context is present — upstream_cluster.name, http.status_code, upstream_address, response_flags, and pod identity (podName, podNamespace) give operators enough information to debug most ingress issues.
  • The Contour configmap's includePodDetail: true option enables operators to add pod identity to spans, showing intentional design for operational usability beyond Envoy defaults.

Opportunities

Add http.route to ingress spans — As an ingress controller, the most valuable single attribute Contour could add is http.route — the matched route pattern (e.g., /api/v1/*). This attribute is what enables route-level RED metrics and is the primary grouping dimension for ingress traffic analysis. Contour knows the matched route at the time of proxying.
Provide a curated SLO metric guide — With 469 Envoy metrics, operators face a discovery problem. A short guide identifying the 10–15 most important metrics for typical ingress SLOs (e.g., envoy_http_downstream_rq_total, envoy_cluster_upstream_rq, envoy_cluster_upstream_rq_time) would dramatically improve the out-of-box experience for new users.
1 Stability & Change Management Tracing documented; no formal telemetry stability contract; changes not explicitly versioned ▶

What we observed

Contour has a dedicated Tracing Support documentation page, a Prometheus metrics guide, and a tracing design document. However, the telemetry behavior itself (span names, attribute names, metric names) is not documented as a contract. Release notes for v1.32.x and v1.33.x contain no dedicated telemetry change sections. Telemetry changes from Envoy upgrades are not explicitly called out. No schemaUrl is set on OTLP trace exports. The Contour deprecation policy covers API and configuration but makes no mention of telemetry signals. Users cannot safely build dashboards and alerts without risk of breakage on upgrade.

What's working well

  • Tracing is documented with a dedicated guide covering configuration options (serviceName, overallSampling, customTags) and a design document explaining the architectural decisions — evidence that telemetry is treated as a user-facing concern.
  • The Prometheus metrics guide provides a solid foundation for metrics observability setup, and the contour_build_info metric provides a stable version anchor that dashboards can reliably use.

Opportunities

Treat telemetry as a versioned contract — Documenting the emitted span names, attribute names, and metric names as a contract — and adding telemetry-specific sections to release notes — would give operators confidence to build dashboards and alerts. Extending the deprecation policy to cover telemetry signals would be a concrete first step.
Set schemaUrl on OTLP trace exports — Currently no schemaUrl is set on OTLP trace exports from Envoy. Setting the schema URL (e.g., https://opentelemetry.io/schemas/1.7.0 for the deprecated attribute set currently in use) would allow tooling to correctly interpret the attribute schema and would make the versioning of the telemetry schema explicit and machine-readable.
Key takeaways

● Strengths

  • Excellent trace correlation — 100% of traffic-generated traces form complete ingress→egress→backend parent-child chains with W3C Trace Context propagated end-to-end. Span kinds are semantically correct. This is the strongest OTel capability Contour offers.
  • Documented tracing architecture — Dedicated tracing documentation page, design document, and configurable options (service name, sampling rate, custom tags, pod detail) show intentional investment in observability as a user-facing feature.
  • Rich Prometheus metrics — 554 unique metric names (14 contour_* control-plane + 469 envoy_* data-plane) provide comprehensive operational visibility. The contour_build_info metric with version label gives dashboards a reliable anchor.

● Priority improvements

  • Update HTTP semantic conventions — Replace deprecated attributes (http.method, http.url, http.status_code as string) with current stable OTel HTTP semconv. Add http.route. Requires Envoy upstream changes, but Contour should advocate for them.
  • Add service.version to trace resource attributes — Version is already available in contour_build_info. Propagating it to trace resource attributes as service.version=v1.32.1 enables version-based filtering and cross-signal correlation with zero new information needed.
  • Enable structured log export — Configuring Envoy access logs in JSON format and collecting via the OTel Collector filelog receiver would add the third signal, enable log-trace correlation, and unlock Level 2 multi-signal observability.
  • Treat telemetry as a versioned contract — Document span names, attribute names, and metric names as a stability contract. Add telemetry-specific changelog entries. Extend the deprecation policy to cover telemetry signals so operators can build reliable dashboards.

● Notable observations

  • Tracing via xDS, not bootstrap — Contour configures Envoy's tracing dynamically via xDS (Listener Discovery Service), not via the Envoy bootstrap JSON. This is a sophisticated pattern enabling runtime reconfiguration, but the Envoy bootstrap file has no tracing section, making the setup non-obvious for users debugging tracing issues.
  • service.name=contour comes from Envoy, not the control plane — Traces with service.name=contour represent Envoy data-plane activity, not the Contour control plane. The control plane emits no OTLP traces at all. This conceptual mismatch can confuse users who expect service.name=contour to reflect control-plane behavior.
  • ExtensionService CRD repurposed for tracing — The ExtensionService CRD (normally for external authorization backends like OPA) is repurposed as the mechanism to reference the OTel Collector endpoint for tracing. This is the only supported configuration path. Users unfamiliar with Contour's CRD ecosystem may find this confusing, and the Bitnami Helm chart uses configmap mode (not CRD mode), requiring awareness of which configuration mechanism is active.
