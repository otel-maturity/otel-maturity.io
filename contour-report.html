Dimension Details
1 Integration Surface OTLP traces working; metrics Prometheus-only; no logs OTel path ▶

What we observed

Contour supports OTLP gRPC trace export via Envoy's native OTel tracer, configured through the ExtensionService CRD and a tracing: block in the Contour ConfigMap. Traces were confirmed flowing — 4 JSONL batches observed in the collector. However, tracing is off by default and requires explicit, project-specific CRD configuration. Metrics are exposed exclusively via Prometheus scrape endpoints: Contour control plane at port 8000 and Envoy data plane at :8002/stats/prometheus. No OTLP metrics path exists. Logs go to stdout only with no structured OTLP export path. Standard OTEL_* environment variables (e.g., OTEL_EXPORTER_OTLP_ENDPOINT) are not respected.

What's working well

  • OTLP gRPC trace export is a first-class, documented feature — OpenTelemetry is explicitly positioned as the supported tracing integration path.
  • The Tracing Support documentation page provides a complete guide for enabling OTel tracing.
  • A rich Prometheus metrics set (655 unique metric names) provides deep visibility into proxy and control-plane behavior.

Opportunities

Add OTLP metrics export — Offering an opt-in OTLP metrics export path (alongside the existing Prometheus endpoint) would enable consistent signal collection, allow service.name to be unified across traces and metrics, and reduce the need for a separate Prometheus scrape pipeline in OTel-native environments.
Support standard OTEL_* environment variables — Respecting OTEL_EXPORTER_OTLP_ENDPOINT, OTEL_SERVICE_NAME, and OTEL_RESOURCE_ATTRIBUTES would let operators configure telemetry using familiar, standard mechanisms rather than Contour-specific ConfigMap fields.
Document the ExtensionService cross-namespace workaround — When the OTel Collector runs in a different namespace (common in production), operators must create a manual ClusterIP Service + Endpoints object in the projectcontour namespace. This workaround is not documented in the tracing guide and is a frequent operator pain point.

Checklist

  • ✅ OTLP is supported for traces
  • ✅ OpenTelemetry is documented as the tracing integration path
  • ❌ OTLP is not the default — tracing is off by default; must be explicitly enabled via CRD
  • ❌ Metrics are Prometheus-only; no OTLP metrics path exists
  • ❌ Logs have no OTLP path — stdout only
  • ❌ OTEL_EXPORTER_OTLP_ENDPOINT and other standard OTel env vars are not respected
  • ❌ Integration surface is signal-inconsistent (OTLP for traces, scrape for metrics, nothing for logs)
0 Semantic Conventions All HTTP span attributes use deprecated OpenTracing-era names; no current OTel semconv in use ▶

What we observed

Every HTTP span attribute emitted by Envoy's OTel tracer uses deprecated pre-1.0 OpenTracing-era attribute names. Not a single current stable OTel semantic convention is used. Key examples: http.method (current: http.request.method), http.status_code (current: http.response.status_code), http.url (current: url.full), http.protocol (current: network.protocol.name + network.protocol.version). Additionally, several attributes are entirely Envoy-proprietary with no OTel mapping: upstream_cluster, upstream_cluster.name, upstream_address, response_flags, node_id, guid:x-request-id, zone. Metric names use Prometheus-style naming (contour_*, envoy_*) rather than OTel conventions. No schemaUrl is set on any trace or metric data from the project.

Important architectural context: Contour's role in tracing is as a configuration layer — it programs Envoy's OTel tracer via xDS. The span attributes and resource attributes are entirely determined by Envoy's built-in OTel tracer implementation, not by Contour code. Improving semconv compliance requires upstream Envoy changes.

What's working well

  • The instrumentation scope does identify itself as envoy, providing clear attribution of the telemetry source.
  • The span data is rich and consistently emitted — the attribute set is comprehensive, even if the naming conventions need updating.

Opportunities

Advocate for / contribute current OTel HTTP semconv in Envoy's OTel tracer — Migrating from http.method → http.request.method, http.status_code → http.response.status_code, http.url → url.full would immediately improve interoperability with OTel-native backends (Jaeger, Grafana Tempo, etc.). This is an upstream Envoy change, but Contour maintainers can advocate, file issues, or contribute.
Add a schemaUrl to trace data — Setting a schema URL at the resource or scope level would allow downstream tooling to understand which version of the OTel semantic conventions the telemetry intends to follow, even if migration is in progress.
Document Envoy-proprietary attributes — Adding a reference page explaining response_flags, upstream_cluster, node_id, and guid:x-request-id values would help operators interpret telemetry without needing deep Envoy expertise.

Checklist

  • ❌ No current OTel HTTP semantic conventions used (http.request.method, url.full, http.response.status_code all absent)
  • ❌ All HTTP attributes use deprecated OpenTracing-era names (http.method, http.status_code, http.url)
  • ❌ Multiple Envoy-proprietary attributes with no OTel mapping (upstream_cluster, response_flags, node_id)
  • ❌ No OTel metric naming conventions — Prometheus-style names only
  • ❌ No schemaUrl on trace or metric data from the project
  • ❌ Operators need Envoy-specific knowledge to interpret most span attributes

2 Trace Modeling & Context Propagation W3C Trace Context propagated correctly in both directions; coherent E2E traces verified across Envoy and backend ▶

What we observed

This is Contour's strongest dimension. Each HTTP request through Envoy produces exactly two spans with a correct parent-child relationship: an ingress span (kind=SERVER) representing the inbound request, and a router … egress span (kind=CLIENT) representing the upstream forwarding. W3C Trace Context propagation was verified in both directions:

  • Without inbound traceparent: Envoy creates a new root ingress span. The backend's GET / span has parentSpanId matching the Envoy egress span — confirming full E2E trace coherence across service boundaries.
  • With inbound traceparent: Envoy's ingress span correctly sets parentSpanId to the caller's span ID, continuing the external trace. Multiple ingress spans all showed the correct traceId and parentSpanId.

Example verified trace tree: ingress (Envoy, SERVER, root) → router demo_otel-eval-backend_3000 egress (Envoy, CLIENT) → GET / (backend, SERVER) → middleware spans.

What's working well

  • W3C Trace Context (traceparent header) is supported and propagated correctly — both inbound (continuing external traces) and outbound (injecting into upstream requests).
  • Entry-point spans are consistently SERVER (kind=2) and upstream forwarding spans are consistently CLIENT (kind=3) — semantically correct per OTel conventions.
  • E2E traces span Envoy and upstream services coherently in a single trace tree — the core value proposition for an ingress controller.
  • The two-span model (ingress + egress) is intentional, lean, and documented.

Opportunities

Improve span names to reflect logical operations — Span names like router demo_otel-eval-backend_3000 egress encode Envoy's internal cluster naming scheme. Adopting OTel HTTP semconv span naming (e.g., GET /path or HTTP GET) would make traces immediately readable to operators unfamiliar with Envoy internals.
Document trace modeling as a design contract — Explicitly documenting the two-span model (what each span represents, what attributes to expect) as a stable contract would give operators confidence to build dashboards and alerts on top of trace data.

1 Multi-Signal Observability Traces (OTLP) and metrics (Prometheus) present but isolated; logs absent; no cross-signal correlation ▶

What we observed

Two of three signals are present: traces flow via OTLP gRPC and metrics are available via Prometheus scrape (655 unique metric names). Logs are absent as an OTLP signal — stdout only. Critically, there is no cross-signal correlation between traces and metrics: traces use service.name=contour while metrics use collector-derived names (service.name=contour-control-plane, service.name=envoy-data-plane). No shared attribute keys exist between trace spans and metric data points. A user cannot pivot from a latency metric to a specific trace, or from a trace to related logs. The mixed push/pull collection model (OTLP push for traces, Prometheus pull for metrics) compounds the correlation challenge.

What's working well

  • Traces are present and flowing via OTLP — a first-class, push-based signal.
  • Metrics are rich and comprehensive: 470 envoy_* metrics, 14 contour_* control-plane metrics, plus Go runtime metrics — covering circuit breakers, upstream health, xDS state, and more.

Opportunities

Unify service identity to enable cross-signal correlation — Aligning service.name across traces and metrics (or documenting the mapping between contour, contour-control-plane, and envoy-data-plane) would allow operators to correlate high-latency traces with relevant metric spikes in Grafana or similar backends.
Add OTLP metrics export — Moving metrics to OTLP push would unify the collection model, allow native service.name to be set consistently, and eliminate the need for a separate Prometheus scrape pipeline.

1 Audience & Signal Quality Lean trace volume (2 spans/request); span names and attributes require Envoy domain knowledge; 655 metrics with no guidance on relevance ▶

What we observed

Trace volume is lean — exactly two spans per request (ingress + router … egress) — which is appropriate for an ingress controller. Sampling is configurable via overallSampling. However, span names encode Envoy's internal cluster naming scheme: router demo_otel-eval-backend_3000 egress encodes namespace (demo), service name (otel-eval-backend), and port (3000). An operator unfamiliar with Envoy's cluster naming convention would need to decode this. Multiple attributes require Envoy-specific knowledge: response_flags uses Envoy's internal flag notation (e.g., - for no flags), upstream_cluster encodes an internal cluster ID (demo/otel-eval-backend/3000/da39a3ee5e), and node_id / guid:x-request-id / zone are Envoy internals. Metrics volume is very high (655 unique names) with no guidance on which are operationally relevant.

What's working well

  • Trace volume is lean (2 spans per request) — no excessive internal spans that would add noise.
  • Sampling is configurable via the overallSampling field, allowing operators to tune trace volume for production use.
  • The customTags and includePodDetail configuration options allow operators to add context to spans beyond the defaults.

Opportunities

Adopt OTel HTTP semconv span naming — Replacing ingress with a name like HTTP GET or {method} {route}, and router demo_otel-eval-backend_3000 egress with a logical upstream service name, would make traces immediately readable without Envoy domain knowledge.
Publish a "top metrics" guide — With 655 unique metric names, operators need guidance on which metrics matter for common scenarios (latency SLOs, error rates, upstream health). A curated list of 10–20 key metrics with descriptions would significantly improve operational usability.
Document Envoy-specific attribute semantics — Adding a reference for response_flags values, upstream_cluster naming format, and other Envoy-internal attributes would reduce the domain knowledge barrier for operators new to Envoy.

Key Takeaways

🟢 Strengths

  • W3C Trace Context propagation is correct and complete — Envoy correctly reads inbound traceparent headers (continuing external traces) and injects traceparent into upstream requests. E2E traces spanning Envoy and backend services are coherent and verified. This is the most important capability for an ingress controller.
  • Rich Envoy metrics set — 470+ envoy_* metrics and 14 contour_* control-plane metrics provide deep visibility into proxy behavior, circuit breakers, upstream health, and xDS control plane state.
  • Tracing is documented and first-class — The Tracing Support page provides a complete guide for enabling OTel tracing via the ExtensionService CRD. OpenTelemetry is explicitly positioned as the supported tracing integration.

🟡 Priority Improvements

  • Adopt current OTel HTTP semantic conventions — All HTTP span attributes use deprecated OpenTracing-era names. Migrating to http.request.method, http.response.status_code, url.full would immediately improve interoperability with OTel-native tooling (requires upstream Envoy changes).
  • Add service.version and respect OTEL_* env vars — The trace resource lacks service.version. Supporting standard OTel env vars for service identity would allow consistent configuration without project-specific ConfigMap changes.
  • Unify service identity across signals — Traces use service.name=contour while metrics use collector-derived names. Consistent identity would enable cross-signal correlation in Grafana or Jaeger.
  • Add OTLP metrics export — Moving metrics to OTLP push would unify the collection model, enable native service.name consistency, and reduce operator setup friction.
  • Introduce telemetry stability documentation — Adding a dedicated "Observability" section to release notes and documenting which attributes/metrics are stable would allow operators to build reliable dashboards without fear of silent breakage.

🔵 Notable Observations

  • Architectural constraint: Envoy owns the trace semantics — Contour's role in tracing is as a configuration layer — it programs Envoy's OTel tracer via xDS. Span attributes, span names, and resource attributes are entirely determined by Envoy's built-in OTel tracer. Improvements to semantic conventions require upstream Envoy changes, which is a significant but not insurmountable constraint.
  • ExtensionService cross-namespace limitation is a real operator pain point — When the OTel Collector runs in a different namespace (common in production), operators must create manual ClusterIP Service + Endpoints objects in the projectcontour namespace. This workaround is not documented in the tracing guide.
  • Envoy metrics path is non-standard (/stats/prometheus) — The Prometheus receiver's default metrics_path=/metrics silently returns 404 for Envoy. This is a common operator mistake that results in no Envoy metrics being scraped, and it is not prominently highlighted in the Prometheus guide.
  • telemetry.sdk.version is a build hash, not a semantic version — The value 3542e3464a2662423065a6ec854905b25955a09e/1.35.10/Clean/RELEASE/BoringSSL is not useful as a version identifier. The actual Envoy version (1.35.10) is embedded but requires parsing to extract.
