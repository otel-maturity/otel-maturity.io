Dimension details
2 Integration Surface OTLP gRPC traces are first-class; metrics are Prometheus-only with no OTLP path; logs are stdout-only ▶

What we observed

Istio 1.29.2 supports native OTLP gRPC trace export via meshConfig.extensionProviders with an opentelemetry provider type, activated per-namespace with a Telemetry CR (telemetry.istio.io/v1). Envoy sidecars and the ingress gateway push spans directly to the collector's OTLP gRPC endpoint (port 4317). 905 spans were confirmed flowing across 405 traces. Metrics, however, are Prometheus-only — scraped from ports 15014 (istiod), 15020, and 15090 (sidecars). No OTLP metrics provider exists in the extension system as of Istio 1.29. Access logs are written to /dev/stdout in JSON format with no OTLP transport.

What's working well

  • OTLP gRPC trace export is a first-class, production-supported integration since Istio 1.13, with a clean Kubernetes-native configuration API via the stable Telemetry CR (v1) and meshConfig.extensionProviders.
  • Sampling is configurable per-namespace via randomSamplingPercentage in the Telemetry CR, and the OTLP endpoint (service + port) is fully configurable in meshConfig.

Opportunities

Add OTLP metrics export — Adding an OTLP metrics provider to meshConfig.extensionProviders (similar to the existing trace provider) would enable a unified collection model and eliminate the need to run separate Prometheus scrapers alongside OTLP receivers.
Enable OTLP log export — Istio's extension provider system supports an envoyFileAccessLog type with OTel formatting. Configuring an OTLP log provider would complete the three-signal picture and enable trace-log correlation via traceId and spanId fields in access logs.

Checklist

  • ✓ OTLP supported for traces (gRPC, port 4317) — 905 spans confirmed flowing
  • ✓ OTLP endpoint is configurable (service + port in meshConfig.extensionProviders)
  • ✗ OTLP not available for metrics (Prometheus scrape only)
  • ✗ OTLP not available for logs (stdout JSON only, no traceId/spanId)
1 Semantic Conventions Envoy spans use deprecated HTTP semconv (http.method, http.status_code, http.url, http.target); Istio-specific attributes are non-standard ▶

What we observed

All Envoy-generated spans use deprecated OpenTelemetry HTTP semantic conventions. Attributes like http.method, http.status_code, http.url, http.target, http.flavor, net.host.ip, and net.peer.ip are present — all deprecated in favor of stable equivalents such as http.request.method, http.response.status_code, url.path, and network.peer.address. This is an upstream Envoy OTel tracer constraint (Envoy 1.37.2-dev). Additionally, Istio adds its own non-standard span attributes: istio.mesh_id, istio.cluster_id, istio.canonical_service, istio.canonical_revision, node_id, upstream_cluster, and response_flags — valuable mesh context, but not mapped to any OTel semconv equivalents. Metric names follow Prometheus conventions (snake_case, _total suffixes) with no OTel metric naming conventions used.

What's working well

  • telemetry.sdk.name (envoy), telemetry.sdk.language (cpp), and telemetry.sdk.version are correctly set on all OTLP trace resources, identifying the instrumentation source.
  • Istio-specific attributes like istio.canonical_service, istio.namespace, istio.mesh_id, and response_flags provide rich mesh-level context that is genuinely useful for operators debugging traffic issues.

Opportunities

Migrate Envoy OTel tracer to current HTTP semconv — Upstream Envoy's OTel tracer currently emits deprecated attributes (http.method → should be http.request.method, http.status_code → http.response.status_code, net.peer.ip → network.peer.address, etc.). Tracking and communicating the upstream Envoy migration timeline would help Istio users plan for the transition.
Map Istio-specific attributes to OTel semconv equivalents — Attributes like istio.namespace could map to k8s.namespace.name, and upstream_cluster context could align with peer.service. Publishing a mapping table between Istio-specific attributes and their nearest OTel semconv equivalents would improve backend portability.
2 Resource Attributes & Configuration Envoy natively emits service.name + telemetry.sdk.* via OTLP; no service.version; not configurable via OTEL_* env vars ▶

What we observed

Envoy's OTLP exporter natively emits a minimal but consistent resource attribute set: service.name (formatted as <canonical_service>.<namespace>, e.g., istio-ingressgateway.istio-ingress and otel-eval-backend.demo), telemetry.sdk.name (envoy), telemetry.sdk.language (cpp), and telemetry.sdk.version. Notably absent natively are service.version, service.namespace (embedded in the service name instead), service.instance.id, and host.name. Istio/Envoy does not support OTEL_* standard environment variables — configuration is exclusively via meshConfig.extensionProviders and the Telemetry CR. The service.name format differs between traces (otel-eval-backend.demo) and metrics (collector job name istio-envoy-demo), preventing cross-signal identity correlation.

What's working well

  • service.name is consistently set on all OTLP trace spans, derived from the pod's service.istio.io/canonical-name label, providing stable workload identity across all spans for the same workload.
  • The telemetry.sdk.* attributes correctly identify the instrumentation (envoy, cpp), enabling backends to distinguish Envoy-generated spans from application-generated spans.

Opportunities

Emit service.version and service.namespace as separate attributes — Currently service.name embeds the namespace (otel-eval-backend.demo), deviating from the OTel standard where service.namespace is a separate attribute. Adding service.namespace and service.version (from the pod's app.kubernetes.io/version label) would improve backend compatibility.
Align service.name format between traces and metrics — Trace spans use service.name=otel-eval-backend.demo while metrics arrive with collector-assigned job names like istio-envoy-demo. Aligning these identities — either by standardizing the service.name format or by using a shared label in the Prometheus scrape configuration — would enable cross-signal correlation without custom pipeline transforms.
3 Trace Modeling & Context Propagation Full W3C traceparent propagation confirmed; correct span kinds; coherent multi-hop traces across gateway and sidecar ▶

What we observed

Istio/Envoy's trace modeling is excellent. Each request generates a coherent 3-layer trace: the ingress gateway emits a root SERVER span followed by a CLIENT egress span, which parents the sidecar's SERVER span, which in turn parents application-level spans. W3C Trace Context (traceparent) propagation was confirmed end-to-end — a test request injected with traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01 was found with the same traceId across ingress gateway spans, sidecar spans, and application spans. Span names follow the pattern <upstream_fqdn>:<port>/* for SERVER spans and router <cluster>; egress for CLIENT spans.

What's working well

  • Full W3C traceparent propagation is honored — externally-injected trace IDs are preserved end-to-end through gateway and sidecar boundaries, enabling integration with upstream tracing systems.
  • Span kinds are semantically correct (SERVER for inbound, CLIENT for outbound routing), and parent-child relationships accurately model the request path: gateway SERVER → gateway CLIENT → sidecar SERVER → app spans.

Opportunities

Document application header forwarding requirement — For cross-service trace continuity, applications must forward incoming traceparent (and b3) headers to outgoing requests — Envoy does not do this automatically. This is a known Istio limitation documented in the observability guide, but more prominent guidance at the SDK/framework level would reduce operator confusion.
Filter low-value infrastructure spans — Spans for internal health checks (GET /health), metadata service egress (egress 169.254.169.254), and cloud metadata endpoints (egress 100.100.100.200) add noise to traces. Providing sampling filter recommendations in the Telemetry CR documentation would help operators reduce noise without losing signal.
1 Multi-Signal Observability Only traces use OTLP; metrics require Prometheus scrape; logs are stdout with no trace context; no cross-signal correlation ▶

What we observed

Only traces use OTLP as the transport. Metrics require a separate Prometheus scrape pipeline (ports 15014, 15020, 15090), and access logs are written to /dev/stdout in JSON with no OTLP transport. Cross-signal correlation is not possible out of the box: access logs contain request_id (an Envoy UUID) but no traceId or spanId; metrics use Prometheus labels (pod, namespace, app) while traces use service.name in a different format. The collection model requires operators to run two separate pipelines — an OTLP receiver for traces and a Prometheus receiver for metrics — with no unified OTLP path for Istio telemetry.

What's working well

  • Traces are fully OTLP-native (gRPC push), providing a solid foundation for the highest-value observability signal in a service mesh — distributed request tracing across gateway and sidecar boundaries.
  • The Prometheus metrics surface is extensive (352 unique metric names) and well-established in the ecosystem, with broad tooling support via Grafana, Prometheus Operator, and existing dashboards.

Opportunities

Add traceId/spanId to access logs for trace-log correlation — Envoy's access log formatter supports %TRACEID% and %REQ(X-B3-TRACEID)% fields. Including traceId and spanId in the JSON access log format would enable correlation between access log entries and distributed traces, significantly improving debugging workflows.
Introduce OTLP metrics export to unify the collection pipeline — A single OTLP pipeline for all three signals would eliminate the split collection model, enable consistent service.name across traces and metrics, and simplify operator configuration. This is the highest-impact improvement for multi-signal observability in Istio.
2 Audience & Signal Quality Trace span names are operator-legible Envoy routing strings; rich Istio metadata; high-value control plane metrics via Prometheus ▶

What we observed

Envoy span names follow the pattern <upstream_fqdn>:<port>/* for SERVER spans (e.g., otel-eval-backend.demo.svc.cluster.local:3000/*) and router outbound|3000||<fqdn>; egress for CLIENT spans — verbose but interpretable by Kubernetes operators. The 905 spans include some noise: internal health check spans (GET /health), Node.js filesystem spans (fs realpathSync, fs readFileSync), and low-value infrastructure spans (egress 169.254.169.254). The 352 unique Prometheus metric names include high-value control plane metrics (pilot_*, citadel_*) alongside a large volume of raw envoy_* internals. Notably, the standard Istio L7 traffic metrics (istio_requests_total, istio_request_duration_milliseconds) were not observed in the collected data.

What's working well

  • Trace span names describe routing context meaningfully — otel-eval-backend.demo.svc.cluster.local:3000/* and router outbound|3000||...; egress are directly useful for operators debugging mesh traffic issues.
  • Istio-specific span attributes (istio.canonical_service, istio.namespace, istio.mesh_id, response_flags, upstream_cluster, guid:x-request-id) provide valuable mesh context beyond standard HTTP attributes, enabling correlation with Envoy access logs via request_id.

Opportunities

Ensure istio_requests_total L7 metrics are flowing — The standard Istio L7 traffic metrics (istio_requests_total, istio_request_duration_milliseconds) — the primary SLI metrics for mesh traffic — were absent from collected data. These require the Envoy stats filter (Telemetry v2 WASM plugin) to be properly configured. Ensuring these metrics are reliably emitted is the highest-priority quality improvement for the metrics signal.
Reduce raw Envoy stats noise in metrics output — The 352 unique metric names include a large volume of raw envoy_cluster_* and istio_agent_* Go runtime metrics that most operators won't need. Providing a recommended metric allowlist or a curated "essential metrics" profile would improve the signal-to-noise ratio for new adopters.
1 Stability & Change Management No schemaUrl on Envoy OTLP traces; Prometheus metrics have no schema URL; telemetry behavior documented but not as a formal contract ▶

What we observed

Istio has documentation for tracing, metrics, and access logs, and the Telemetry CR API (telemetry.istio.io/v1) was promoted to stable (v1) in Istio 1.23. However, the telemetry output itself (span attributes, metric names, metric labels) has no schema versioning — no schemaUrl is set on Envoy OTLP trace exports at either the resource or scope level, and Prometheus-scraped metrics carry no schemaUrl. The telemetry output format is subject to change with Envoy version upgrades, and breaking changes can occur without explicit versioned notice. The deprecated HTTP semconv attributes currently in use are a concrete example of a known migration that has not yet occurred.

What's working well

  • The Telemetry CR configuration API (telemetry.istio.io/v1) is stable, providing a reliable and versioned configuration surface for operators to configure tracing, metrics, and logging behavior.
  • Telemetry changes are communicated in Istio release notes (e.g., 1.29 release notes), and the documentation covers tracing, metrics, and access log configuration at istio.io/latest/docs/tasks/observability/.

Opportunities

Set schemaUrl on Envoy OTLP trace exports — Adding a schemaUrl (e.g., https://opentelemetry.io/schemas/1.18.0) to Envoy's OTLP resource and scope exports would enable observability backends and the OTel Collector's transform processor to understand which semconv version is in use, and would signal to users when the deprecated attributes will be migrated.
Publish a telemetry stability contract separate from the configuration API — The Telemetry CR API is stable, but the telemetry output (span attributes, metric names, label cardinality) is not versioned independently. Documenting which attributes and metric names are considered stable output vs. implementation details — and communicating breaking changes with advance notice — would significantly improve the operator experience across Envoy version upgrades.
Key takeaways

● Strengths

  • W3C Trace Context propagation — Istio/Envoy correctly honors externally-injected traceparent headers and propagates trace context across gateway and sidecar boundaries, generating coherent multi-hop traces confirmed across 405 unique trace IDs.
  • Native OTLP gRPC trace export — The Telemetry CR + meshConfig.extensionProviders provides a clean, Kubernetes-native configuration API. The Telemetry CR is stable (v1), sampling is per-namespace configurable, and the integration requires no sidecar code changes.
  • Rich operational span attributes — Envoy spans include istio.canonical_service, istio.namespace, istio.mesh_id, response_flags, upstream_cluster, and guid:x-request-id — valuable mesh-level context for debugging traffic issues even without full semconv alignment.

● Priority improvements

  • Migrate to current HTTP semconv — Replace deprecated attributes (http.method, http.status_code, net.*) with stable equivalents (http.request.method, http.response.status_code, network.*) — requires upstream Envoy OTel tracer changes.
  • Add OTLP metrics export — Extend meshConfig.extensionProviders with an OTLP metrics provider to unify the collection pipeline and enable consistent service.name identity across traces and metrics.
  • Add traceId/spanId to access logs — Use Envoy's %TRACEID% access log formatter to include trace context in JSON access logs, enabling correlation between log entries and distributed traces.
  • Set schemaUrl on OTLP exports — Add schemaUrl to Envoy OTLP resource and scope exports to enable tooling to understand the semconv version in use and support automated migration transforms.

● Notable observations

  • istio_requests_total was absent — The primary Istio L7 SLI metrics (istio_requests_total, istio_request_duration_milliseconds) were not present in scraped data. These require the Envoy stats filter (Telemetry v2 WASM plugin) — the mesh-metrics Telemetry CR was applied but the metrics did not appear on ports 15020 or 15090 for the demo workload. This may be a configuration timing issue.
  • Non-standard service.name format embeds namespace — Envoy sets service.name to otel-eval-backend.demo (embedding the namespace), deviating from the OTel standard of a separate service.namespace attribute. This makes it harder to use service.name as a direct workload identifier in backends that expect standard format.
  • telemetry.sdk.version is a git hash, not semver — The value af30be60b7c35f2aceaea1b7382c7fbf12aa5e67/1.37.2-dev/Clean/RELEASE/BoringSSL is an Envoy build identifier, not a standard version string. This makes it difficult to correlate the SDK version to a specific Envoy release or track semconv migration progress in automated tooling.
