Dimension details
1 Integration Surface OTLP tracing driver exists but is WIP; metrics only via Prometheus scrape; no OTLP log export ▶

What we observed

Emissary-ingress v3.12.2 supports three tracing drivers via the TracingService CRD: zipkin, datadog, and opentelemetry. The opentelemetry driver generates a correct Envoy bootstrap config pointing to port 4317 via gRPC, but the HTTP Connection Manager (HCM) tracing config in each listener remains empty {} — so no spans are emitted and no trace context is propagated. The tracing cluster shows cx_total: 0. Metrics are exposed exclusively via Prometheus scrape on admin port 8877 (/metrics). Logs use Envoy's proprietary ALS gRPC protocol, which is incompatible with the standard OTLP receiver. No OTEL_* environment variables are supported; all configuration is CRD-based. Notably, v4.0.1 (post-evaluation) includes a working OTel distributed tracing how-to, suggesting the driver was fixed in v4.

What's working well

  • The TracingService CRD with driver: opentelemetry is correctly designed — the Envoy bootstrap config is generated with the right gRPC cluster, W3C Trace Context propagation mode, and 128-bit trace IDs. The infrastructure is in place; only the HCM hook is missing.
  • Prometheus metrics on port 8877 are reliably scraped and converted to OTLP by the OTel Collector's Prometheus receiver, providing a working path for operational monitoring today.

Opportunities

Complete the HCM tracing hook — Enabling the opentelemetry driver to populate the HTTP Connection Manager tracing config would immediately unlock span emission and W3C Trace Context propagation. This appears to be a targeted, fixable gap — confirmed fixed in v4.
Add OTLP metrics push and document OTEL_* support — Adding native OTLP metrics emission alongside Prometheus scraping would enable a unified pipeline. Supporting OTEL_EXPORTER_OTLP_ENDPOINT and OTEL_SERVICE_NAME would allow users to plug Emissary into any OTel-compatible backend without a Prometheus receiver.

Checklist

  • ✓ OTLP tracing driver (driver: opentelemetry) exists in the TracingService CRD
  • ✓ Prometheus metrics are reliably accessible and consumable via OTel Collector Prometheus receiver
  • ✗ OTLP is not the default or recommended export path — Prometheus scraping is the only working option
  • ✗ Standard OTEL_* environment variables are not supported; configuration is CRD-only
1 Semantic Conventions Envoy Prometheus metrics use proprietary envoy_* naming; no OTel semconv alignment; traces use deprecated HTTP attributes ▶

What we observed

Emissary's 407 Prometheus metric names follow Envoy's internal naming conventions: envoy_cluster_*, envoy_http_*, envoy_listener_*, envoy_server_*, and ambassador_*. Data point labels use Envoy-internal names like envoy_cluster_name, envoy_http_conn_manager_prefix, envoy_response_code_class, and ambassador_id — none of which correspond to OTel semantic conventions. Since Emissary emits no spans itself, trace semconv alignment cannot be assessed for Emissary directly. Backend traces observed in the evaluation used deprecated HTTP attributes: http.method, http.status_code, http.url, http.target, http.host, http.flavor, net.host.name, net.peer.ip, and net.transport — all superseded in OTel HTTP semconv v1.23+.

What's working well

  • The TracingService CRD references W3C Trace Context (TRACE_CONTEXT) and B3 propagation modes by their correct OTel names, showing awareness of the OTel propagation ecosystem.
  • The ambassador_* metrics (ambassador_aconf_time_seconds, ambassador_process_cpu_seconds_total, ambassador_process_resident_memory_bytes) follow Prometheus naming conventions and provide useful process-level signals.

Opportunities

Introduce an OTLP metrics path with OTel HTTP semconv names — When adding OTLP metrics export, consider mapping key Envoy stats to current OTel HTTP semantic conventions: http.server.request.duration, http.server.active_requests, http.response.status_code. This would enable off-the-shelf OTel dashboards and alerts without Envoy-specific knowledge.
Align trace attributes with current OTel HTTP semconv when tracing is enabled — Once the OTel tracing driver is functional, ensure spans use current conventions: http.request.method (not http.method), http.response.status_code (not http.status_code), url.full (not http.url), server.address (not net.host.name).
1 Resource Attributes & Configuration No native service.name from Emissary itself; identity is pipeline-derived; no OTEL_* env var support ▶

What we observed

Emissary does not use the OTel SDK and sets no resource attributes natively. The service.name: emissary-ingress visible in metrics is set by the OTel Collector's Prometheus receiver from the scrape job's job_name — it is pipeline-derived, not emitted by Emissary. The TracingService CRD has a config.service_name field, which would set the service name in Envoy's bootstrap tracing config, but since the OTel driver is WIP this is never observed in practice. No OTEL_EXPORTER_OTLP_ENDPOINT, OTEL_SERVICE_NAME, or OTEL_RESOURCE_ATTRIBUTES environment variables are supported. Kubernetes metadata (k8s.pod.uid, k8s.namespace.name, k8s.deployment.name) was added by the OTel Collector's k8sattributes processor for k8s_cluster receiver metrics, but not for Prometheus-scraped metrics (due to Service DNS name vs. pod IP association).

What's working well

  • The TracingService CRD's config.service_name field shows awareness of service identity — once the OTel tracing driver is functional, this field would set service.name in Envoy's bootstrap config.
  • The Kubernetes-native CRD configuration model integrates well with GitOps workflows, and the OTel Collector's k8sattributes processor can enrich metrics with k8s.deployment.name, k8s.namespace.name, and container metadata.

Opportunities

Emit service.name, service.version, and service.instance.id natively — Adding OTel SDK resource detection (or equivalent) so that Emissary itself sets service.name, service.version: 3.12.2, and service.instance.id would ensure consistent identity across all signals without relying on pipeline configuration.
Support OTEL_* environment variables for configuration — Exposing OTEL_EXPORTER_OTLP_ENDPOINT and OTEL_RESOURCE_ATTRIBUTES alongside the existing CRD-based model would allow operators to configure telemetry using standard OTel tooling and follow the OTel specification's configuration contract.
0 Trace Modeling & Context Propagation Emissary emits no spans in v3.12.2 (OTel driver WIP); no trace context propagation to upstream services ▶

What we observed

Emissary emits no spans in v3.12.2. The opentelemetry TracingService driver generates a correct Envoy bootstrap config (gRPC cluster pointing to port 4317 with envoy.opentelemetry), but the HTTP Connection Manager tracing config in each listener is empty {}. As a result, the tracing cluster shows cx_total: 0 and no tracing.* stats appear. Context propagation was explicitly tested: requests sent with traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01 arrived at the backend with parentSpanId: null — confirming Emissary did not forward the header. The TracingService was configured with propagation_modes: [TRACE_CONTEXT, B3], but since the HCM tracing hook is inactive, no propagation occurs. This is tracked as GitHub issue #5573 (opened 2024-02-15, last updated 2026-04-14).

What's working well

  • The tracing architecture is sound: the TracingService CRD correctly specifies W3C Trace Context and B3 propagation modes, 128-bit trace IDs, and gRPC export — the design is right, only the HCM activation is missing.
  • The issue is publicly tracked and acknowledged (#5573), and v4.0.1 documentation confirms a working OTel tracing integration, showing the project is actively progressing.

Opportunities

Enable the HCM tracing hook for the opentelemetry driver — Populating the HTTP Connection Manager tracing configuration when driver: opentelemetry is set would immediately enable span emission and W3C Trace Context propagation to upstream services. This is the single highest-impact improvement for this dimension.
Backport documentation of the WIP limitation to v3.x — Until users upgrade to v4, clearly documenting that driver: opentelemetry is non-functional in v3.x (and providing a migration guide to v4) would prevent misconfiguration and confusion.
0 Multi-Signal Observability Only Prometheus metrics are effectively usable; traces absent; logs absent; no cross-signal correlation ▶

What we observed

Only one of three signals is effectively usable in v3.12.2. Metrics flow via Prometheus scrape (407 metric types from admin port 8877). Traces do not flow — the OTel driver is WIP. Logs do not flow — the LogService CRD uses Envoy's gRPC Access Log Service (ALS) protocol, which is incompatible with the standard OTLP receiver. No cross-signal correlation is possible: metrics carry no trace IDs or span IDs, there are no logs, and backend traces have no connection to Emissary metrics. The collection models also differ per signal: Prometheus pull for metrics, OTLP gRPC push (WIP) for traces, and Envoy ALS gRPC (protocol mismatch) for logs — requiring three different receiver configurations.

What's working well

  • Prometheus metrics provide comprehensive operational coverage today — 407 metric types covering envoy_cluster_*, envoy_http_*, envoy_listener_*, and ambassador_* signals. Standard Envoy dashboards (e.g., Grafana library) can be used without custom configuration.
  • The project has CRD abstractions for all three signal types (TracingService, LogService, admin service for metrics), showing architectural intent to support multi-signal observability even if the implementations are at different maturity levels.

Opportunities

Enable traces (fixes tracing driver) to unlock two-signal observability — Once the OTel tracing driver is functional, operators will be able to correlate aggregate metrics with individual request traces. Adding trace IDs to metrics (exemplars) would further enable pivoting from a slow-request metric to the specific trace.
Add OTLP log export to replace or complement ALS — The LogService CRD's Envoy ALS integration could be complemented with an OTLP log export path. Once tracing is working, structured access logs emitted as OTLP log records with trace_id and span_id would enable true three-signal correlation.
1 Audience & Signal Quality 407 Prometheus metric types provide operational coverage; naming is Envoy-internal; no user-oriented signal design ▶

What we observed

The 407 Prometheus metric types are comprehensive but primarily expose Envoy's internal stats. Useful operational metrics include envoy_http_downstream_rq_total, envoy_cluster_upstream_rq_total, and envoy_cluster_circuit_breakers_default_cx_pool_open. However, many are highly internal: envoy_cluster_http2_inbound_empty_frames_flood, envoy_cluster_http2_inbound_priority_frames_flood, envoy_cluster_http2_inbound_window_update_frames_flood. The ambassador_* process-level metrics (ambassador_aconf_time_seconds, ambassador_process_cpu_seconds_total, ambassador_process_resident_memory_bytes) are useful and well-named. Emissary emits no spans, so span naming quality cannot be assessed. Without tracing, operators can monitor aggregate health but cannot investigate individual failing requests.

What's working well

  • The 407 Prometheus metrics support basic monitoring and alerting without customization — standard Envoy dashboards exist in the Grafana dashboard library, giving operators an immediate starting point.
  • The ambassador_* metrics provide clean process-level visibility into config processing time, CPU, and memory — these are well-suited for SLO-based monitoring.

Opportunities

Design user-oriented signal layers alongside raw Envoy stats — Consider exposing a curated set of high-level OTel metrics (request rate, error rate, latency percentiles with OTel semconv names) in addition to the full Envoy stat dump. This would serve operators who don't need deep Envoy internals and enable off-the-shelf OTel dashboards.
Enable traces to support root-cause investigation — The most impactful quality improvement is enabling the OTel tracing driver. With traces flowing, operators can move from aggregate metrics (e.g., elevated envoy_cluster_upstream_rq_5xx) to specific failing request traces, dramatically improving mean-time-to-resolution.
1 Stability & Change Management OTel driver WIP status disclosed in logs but not docs; telemetry changes not treated as a contract; no schema URL ▶

What we observed

The opentelemetry driver's WIP status is disclosed in Emissary's runtime logs ("The OpenTelemetry tracing driver is work-in-progress. Functionality is incomplete and it is not intended for production use.") but is not reflected in the TracingService reference documentation or the CRD schema. The v3.9 CHANGELOG presented the OTel driver as a feature, not as experimental. Major telemetry changes (LightStep driver removal, OTel driver introduction) are documented in the CHANGELOG. No schema URL is set on Emissary-emitted telemetry — the schemaUrl: https://opentelemetry.io/schemas/1.18.0 visible on some metrics was set by the OTel Collector, not by Emissary. The TracingService CRD has an API version inconsistency: the propagation_modes field name differs between v3alpha1 and v1 (v3PropagationModes), handled transparently by webhook conversion.

What's working well

  • Major telemetry changes (LightStep driver removal, OTel driver introduction, Zipkin HTTP_JSON_V1 deprecation) are documented in the CHANGELOG, showing some commitment to communicating telemetry evolution.
  • The WIP disclosure in runtime logs is better than silence — operators who check logs will see the warning before investing time debugging a non-functional integration.

Opportunities

Surface the WIP status prominently in documentation and CRD schema — Adding a description annotation to the driver: opentelemetry option in the CRD schema, and a prominent callout in the TracingService docs, would prevent v3.x users from silently misconfiguring a non-functional integration.
Treat telemetry as a versioned contract with schema URLs — Setting schemaUrl on Emissary-emitted OTLP telemetry (once OTLP export is functional) and distinguishing stable from experimental telemetry in release notes would give operators confidence that telemetry won't break unexpectedly across upgrades.
Key takeaways

● Strengths

  • Rich Prometheus metrics coverage — 407 unique Envoy metric types provide comprehensive operational visibility into cluster, HTTP, listener, server, and TLS behavior. Community dashboards (Grafana library) are available out-of-the-box.
  • OTel tracing architecture is in place — The TracingService CRD with driver: opentelemetry, W3C Trace Context propagation, 128-bit trace IDs, and gRPC export is correctly designed. The infrastructure is ready; only the HCM hook is missing — a fixable gap, not a fundamental design problem.
  • v4 shows significant improvement — The v4.0.1 release includes a documented, working "Distributed Tracing with OpenTelemetry and Dash0" integration, confirming the project is actively progressing toward OTel maturity.

● Priority improvements

  • Fix the OTel tracing driver (critical) — Populating the HCM tracing config when driver: opentelemetry is set is the single highest-impact improvement. This unlocks span emission, W3C Trace Context propagation, and multi-signal observability in one change.
  • Add OTLP metrics push with native resource attributes — Supporting OTLP metrics export with service.name, service.version, and service.instance.id set natively would enable a unified OTel pipeline and eliminate the need for a Prometheus receiver.
  • Implement OTLP log export — Complementing the LogService CRD's ALS integration with an OTLP log export path, with trace context fields once tracing is working, would enable true three-signal correlation.
  • Document WIP status prominently — Adding a CRD schema description annotation and a docs callout for driver: opentelemetry in v3.x would prevent silent misconfiguration for users who haven't yet upgraded to v4.

● Notable observations

  • WIP driver is a surprisingly small gap — v3.12.2 has all the OTel tracing infrastructure except the final HCM hook. The bootstrap config is correct, the CRD is correct, the cluster is defined — only the per-listener tracing activation is missing. The v4 fix confirms this was a targeted, surgical change.
  • Prometheus receiver auto-sets service.name — creating an illusion — The OTel Collector's Prometheus receiver sets service.name: emissary-ingress from the scrape job's job_name. This looks like proper resource attribution but is entirely pipeline-derived. Emissary itself sets no resource attributes.
  • k8sattributes enrichment did not apply to Prometheus metrics — Because the Prometheus receiver scrapes a Service DNS name (not a pod IP), the k8sattributes processor could not associate the scrape target with a pod. Operators who want Kubernetes metadata on Prometheus metrics need to configure pod-IP-based scraping or add static resource attributes in the collector pipeline.
