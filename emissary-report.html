Dimension details
1 Integration Surface OTLP driver exists but is non-functional; working path uses Zipkin protocol via a separate CRD ▶

What we observed

Emissary provides a TracingService CRD that nominally supports an opentelemetry driver (OTLP gRPC) alongside zipkin, lightstep, and datadog. However, the opentelemetry driver is explicitly marked "work-in-progress" and does not function in practice — it logs a warning and produces no spans. The only working trace export path is via the zipkin driver, which sends Zipkin v2 JSON over HTTP to port 9411. Metrics are exposed only via a Prometheus endpoint at :8877/metrics. Configuration is entirely through Emissary-specific CRDs and Helm chart values, not through OTEL_* environment variables.

What's working well

  • The TracingService CRD schema explicitly references opentelemetry as a driver option, demonstrating intent to support OTLP natively — the architectural groundwork is in place.
  • The Zipkin driver reliably exports traces to an OTel Collector configured with a Zipkin receiver, providing a functional bridge into OTel pipelines today. Prometheus scraping is well-documented and widely compatible.

Opportunities

Fix or complete the OTLP driver — The opentelemetry driver in TracingService should either be completed to configure Envoy's OTLP tracing bootstrap correctly, or clearly marked as unsupported in documentation. This single change would be the highest-impact improvement for integration surface maturity.
Support OTEL_* environment variables — Adding support for standard OTEL_EXPORTER_OTLP_ENDPOINT, OTEL_SERVICE_NAME, and related env vars would allow operators to configure telemetry using the same patterns as any other OTel-instrumented service, dramatically reducing onboarding friction.

Checklist

  • ✓ OTLP is nominally supported — the opentelemetry driver field exists in the CRD schema and changelog
  • ✓ Zipkin export works as an alternative path to an OTel Collector; Prometheus scraping is well-supported
  • ✗ The opentelemetry driver is non-functional in v3.12.2 — it logs "work-in-progress" and produces no spans
  • ✗ Standard OTEL_* environment variables are not respected; users cannot connect to an OTel Collector without project-specific knowledge
0 Semantic Conventions Deprecated HTTP attributes throughout; proprietary Envoy-internal attribute names; no OTel semconv alignment ▶

What we observed

Emissary (via Envoy's Zipkin tracing) emits span attributes that are deprecated or non-standard relative to current OTel semantic conventions. The ingress span uses http.method (deprecated — current: http.request.method), http.status_code (deprecated — current: http.response.status_code), http.url (deprecated — current: url.full), and peer.address (non-standard — current: network.peer.address). Additionally, Envoy-internal attributes like downstream_cluster, node_id, response_flags, guid:x-request-id, upstream_cluster, and upstream_cluster.name have no OTel equivalent. Both spans use kind=3 (CLIENT) — the ingress span should be kind=2 (SERVER). The instrumentation scope name and version are empty. Metric names use envoy_* and ambassador_* prefixes with no OTel semconv alignment.

What's working well

  • Trace attributes cover the key HTTP dimensions (method, status code, URL, request/response size), giving operators the core data needed for request debugging — the naming just needs updating to current conventions.
  • The metrics set includes operationally meaningful signals like envoy_cluster_upstream_rq_total, envoy_http_downstream_rq_2xx, and ambassador_reconfiguration_time_seconds — the data is present even if the naming differs from OTel semconv.

Opportunities

Align HTTP span attributes with current semconv — Updating to http.request.method, http.response.status_code, url.full, network.protocol.name, network.peer.address, and fixing the ingress span kind to SERVER would unlock compatibility with off-the-shelf OTel dashboards and alert rules without any user configuration.
Set a non-empty instrumentation scope name — Adding a scope name (e.g., io.emissary-ingress.envoy) and version to the instrumentation scope would allow observability tools to identify the source of spans and apply version-specific processing rules.
0 Resource Attributes & Configuration No service.version; service.name differs between traces and metrics; no OTEL_* env var support ▶

What we observed

Emissary natively emits only one resource attribute on trace data: service.name: ambassador-emissary, derived from the Envoy node cluster name. For metrics, service.name: emissary is set by the OTel Collector's Prometheus receiver — it is not emitted by Emissary itself. This creates a critical inconsistency: the same pod appears as ambassador-emissary in traces and emissary in metrics, making cross-signal correlation by service identity impossible. No service.version, service.instance.id, service.namespace, or telemetry.sdk.* attributes are emitted natively. Standard OTEL_SERVICE_NAME and OTEL_RESOURCE_ATTRIBUTES environment variables have no effect since Emissary does not use the OTel SDK.

What's working well

  • After OTel Collector enrichment via the k8sattributes processor, traces gain useful Kubernetes context: k8s.namespace.name, k8s.pod.name, k8s.deployment.name, k8s.node.name, and pod labels including helm.sh/chart: emissary-ingress-8.12.2.
  • The service.name value ambassador-emissary is consistent within the traces signal, enabling trace-level service grouping even if cross-signal correlation is not yet possible.

Opportunities

Unify service.name across all signals — Using the same service.name value in both traces and metrics (e.g., always emissary-ingress) would immediately enable cross-signal correlation in tools like Grafana and Jaeger without any collector-side transforms.
Emit service.version natively — Including the Emissary version (e.g., service.version: 3.12.2) as a resource attribute in trace data would enable version-based analysis, making it possible to compare telemetry before and after upgrades automatically.
1 Trace Modeling & Context Propagation Coherent parent-child structure within Emissary, but B3 propagation prevents end-to-end correlation with W3C-native downstream services ▶

What we observed

Emissary produces two spans per request with a correct parent-child relationship: an ingress span (localhost:8090) and an egress span (router otel_eval_backend_demo_svc_cluster_local_3000 egress). However, Envoy's Zipkin driver injects B3 headers (x-b3-traceid, x-b3-spanid, x-b3-sampled) into upstream requests, while modern OTel-instrumented services use W3C Trace Context (traceparent). This creates a hard propagation boundary: analysis of trace IDs confirmed zero overlap between Emissary traces and backend traces across 6 requests — every request results in two completely separate, uncorrelated trace trees. Both spans also use kind=3 (CLIENT) when the ingress span should be kind=2 (SERVER).

What's working well

  • Within Emissary's own spans, the parent-child relationship is correct and coherent — the egress span is properly nested under the ingress span, making it possible to see the full request lifecycle within Emissary itself.
  • Sampling is configurable via the TracingService CRD, and 100% sampling worked correctly during the evaluation, confirming the tracing pipeline is operational end-to-end.

Opportunities

Add W3C Trace Context propagation support — Enabling dual B3+W3C propagation (or switching to W3C-only) in the TracingService propagation_modes field would allow Emissary to join traces with OTel-native downstream services, enabling true end-to-end distributed tracing. The propagation_modes field already exists in the CRD — this may just require documentation and testing.
Fix ingress span kind to SERVER — The span that receives inbound HTTP traffic should use SPAN_KIND_SERVER rather than SPAN_KIND_CLIENT. This is a semantic correctness fix that would improve compatibility with tools that use span kind for service topology inference.
1 Multi-Signal Observability Traces and metrics both flow; logs are stdout-only with no OTLP export; no cross-signal correlation ▶

What we observed

Two of three signals are flowing: traces via Zipkin v2 JSON HTTP and metrics via Prometheus scrape at :8877/metrics. Logs are not available via OTLP — Envoy access logs go to stdout in text format only. The LogService CRD can forward to a gRPC access log service, but this is not OTLP. Cross-signal correlation is not possible: service.name differs between traces (ambassador-emissary) and metrics (emissary), and stdout logs contain no trace or span IDs. Emissary traces and backend traces are completely separate due to the B3/W3C propagation mismatch (confirmed: 0 shared trace IDs across 6 requests).

What's working well

  • Two signals (traces and metrics) are both available and flowing through an OTel Collector pipeline simultaneously, which exceeds Level 0 and provides meaningful operational coverage for request monitoring and routing analysis.
  • Both signals are compatible with standard OTel Collector receivers (Zipkin receiver for traces, Prometheus receiver for metrics), meaning they can be integrated into existing OTel pipelines without custom adapters or plugins.

Opportunities

Unify service.name to enable trace-metric correlation — Using the same service.name in both traces and metrics would allow tools like Grafana to automatically link a trace's service to its corresponding metrics dashboard, unlocking multi-signal investigation workflows.
Add structured log export with trace context — Implementing OTLP log export (or at minimum enriching access logs with trace_id and span_id fields) would complete the three-signal picture and enable log-to-trace correlation during incident investigation.
1 Audience & Signal Quality Span names are host:port strings and internal cluster identifiers; metrics are comprehensive but use Envoy-internal naming ▶

What we observed

Emissary span names reflect internal Envoy infrastructure rather than logical operations. The ingress span is named localhost:8090 (the listener address during testing — in production this would be the listener IP:port), and the egress span is named router otel_eval_backend_demo_svc_cluster_local_3000 egress (the full Kubernetes DNS cluster name). These names would not be usable with off-the-shelf dashboards or alert rules. User-oriented names would be something like HTTP GET / and upstream otel-eval-backend. On the metrics side, 443 unique metric names are scraped, including many low-level Envoy internals that will be zero in most deployments. The integration also has several undocumented gotchas that reduce default usability: createDefaultListeners: true must be set explicitly, the opentelemetry driver is non-functional, and config.service_name in ZipkinConfig causes a crash loop.

What's working well

  • The metrics set includes genuinely useful operational signals: envoy_cluster_upstream_rq_total, envoy_http_downstream_rq_2xx, envoy_tracing_zipkin_spans_sent, ambassador_reconfiguration_time_seconds, and ambassador_process_resident_memory_bytes — the data for effective monitoring is present.
  • Trace attributes cover the key HTTP dimensions needed for request debugging: method, status code, URL, request and response sizes, user agent, and upstream address — useful for diagnosing routing issues even with deprecated naming.

Opportunities

Use logical operation names for spans — Replacing listener-address-based span names with HTTP method + path (e.g., HTTP GET /api/v1/resource for the ingress span) and service-name-based names (e.g., upstream backend-service for the egress span) would make traces immediately useful in tools like Jaeger, Tempo, and Honeycomb without any post-processing.
Document known gotchas and provide a working quickstart — Prominently documenting the createDefaultListeners: true requirement, the non-functional opentelemetry driver status, and the config.service_name crash would save operators significant debugging time and improve the out-of-box experience substantially.
1 Stability & Change Management Some telemetry changes appear in release notes; no formal telemetry stability contract; opentelemetry driver documented but non-functional ▶

What we observed

Emissary documents tracing via the TracingService CRD reference and a tracing guide. The changelog references telemetry-related changes: the addition of the opentelemetry driver (noted as experimental), support for custom_tags and propagation_modes in TracingService, and the removal of the extra metrics endpoint as a breaking change. However, the opentelemetry driver's non-functional state in v3.12.2 is not clearly documented. Several breaking behaviors were discovered during installation that are not documented anywhere: config.service_name in ZipkinConfig causes an Envoy crash loop, createDefaultListeners: true is required for Envoy to bind HTTP port, and TracingService changes require a full pod restart (bootstrap config, not xDS). No schema URL is set on any telemetry signal. There is no formal stability/experimental distinction for telemetry signals.

What's working well

  • Telemetry-related changes are included in the CHANGELOG, including breaking changes to telemetry-adjacent features (e.g., metrics endpoint removal) — demonstrating awareness that these changes have operational impact on users.
  • The TracingService CRD reference documentation exists and covers the key configuration fields, giving operators a starting point for configuration even if some behaviors are undocumented.

Opportunities

Document known limitations and breaking behaviors — Adding a troubleshooting section to the tracing guide covering the non-functional opentelemetry driver, the config.service_name crash, the createDefaultListeners requirement, and the pod restart requirement for TracingService changes would prevent significant operator frustration and reduce support burden.
Add a schema URL and formal telemetry stability policy — Setting a schemaUrl on telemetry signals (e.g., https://opentelemetry.io/schemas/1.23.0) and publishing a stability policy distinguishing stable from experimental telemetry would allow operators to build long-lived dashboards and alerts with confidence that signals won't silently change.
Key takeaways

● Strengths

  • Rich Prometheus metrics coverage — 443 unique metric names covering Envoy cluster health, HTTP request/response rates, circuit breaker states, and Ambassador control-plane timing provide a comprehensive operational dataset.
  • Functional trace export path — Despite the OTLP driver being non-functional, the Zipkin driver reliably exports traces to an OTel Collector, enabling trace-based debugging of request routing and upstream behavior today.
  • OTel Collector compatibility — Emissary's Zipkin and Prometheus outputs are compatible with standard OTel Collector receivers, allowing integration into existing OTel pipelines without custom adapters.

● Priority improvements

  • Fix or remove the opentelemetry TracingService driver — The OTLP gRPC driver is listed in the CRD schema but does not function in v3.12.2. Fixing it would be the single highest-impact improvement for integration surface maturity.
  • Align span attributes with current OTel HTTP semconv — Replace deprecated attributes (http.method → http.request.method, http.status_code → http.response.status_code, http.url → url.full) and fix span kinds.
  • Unify service.name across traces and metrics — The mismatch between ambassador-emissary (traces) and emissary (metrics) prevents cross-signal correlation without custom pipeline transforms.
  • Add W3C Trace Context propagation — Enabling traceparent injection alongside B3 would allow Emissary to join traces with OTel-native downstream services, enabling true end-to-end distributed tracing.

● Notable observations

  • Non-functional OTLP driver is the most impactful finding — Users who configure driver: opentelemetry in TracingService will get no traces. The driver logs "work-in-progress" and has been in this state since Envoy 1.24 without resolution in v3.12.2.
  • config.service_name causes a crash loop — Including config.service_name in ZipkinConfig causes Emissary pods to crash-loop due to a bug in Emissary's config generation that Envoy 1.31.4 rejects. This is undocumented and will surprise operators.
  • B3 propagation creates a hard tracing boundary — Because Envoy's Zipkin driver injects B3 headers while modern OTel services use W3C Trace Context, every request through Emissary creates two completely separate trace trees. Zero shared trace IDs were observed across 6 requests — this is a fundamental limitation of the Zipkin driver path.
