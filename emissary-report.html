Dimension Details
1 Integration Surface Traces via Zipkin (not OTLP); metrics via Prometheus scrape only; no log export ▶

What we observed

Emissary-Ingress relies on Envoy's built-in tracing subsystem rather than an OTel SDK. Traces are emitted in Zipkin HTTP_JSON format (driver: zipkin) to the OTel Collector's Zipkin receiver on port 9411, which then converts them to OTLP internally. The TracingService CRD does expose a driver: opentelemetry option, but this is silently broken in v3.12.2: the Python config generator (v3tracing.py) produces the Envoy tracer name "envoy.opentelemetry" instead of the correct "envoy.tracers.opentelemetry", causing Envoy to silently ignore the tracer configuration entirely — confirmed via Envoy admin API (zero connections established). Metrics are exposed as ~413 Prometheus-format counters/gauges/histograms at port 8877 (/metrics) and must be scraped externally. No log export path exists beyond stdout plain text.

What's working well

  • Traces are collectable with reasonable effort using the Zipkin workaround and an OTel Collector.
  • The TracingService CRD provides a documented, Kubernetes-native configuration surface for tracing.
  • Prometheus scrape annotations are included in the Helm chart, making metrics collection straightforward for teams already using Prometheus.

Opportunities

Fix the driver: opentelemetry bug — A one-line change in python/ambassador/envoy/v3/v3tracing.py to produce "envoy.tracers.opentelemetry" instead of "envoy.opentelemetry" would enable native OTLP gRPC trace export, eliminating the Zipkin workaround entirely and immediately elevating this dimension.
Add OTLP metrics push support — Envoy supports OTLP stats sinks natively. Exposing this through the Helm chart or a CRD would allow operators to push metrics via OTLP rather than requiring a pull-based Prometheus scrape, enabling a unified OTLP collection path.
Structured log export — Emissary already supports JSON access log format via the Module CRD. Adding an OTLP log export path (or at minimum documenting a collector-based log collection approach) would complete the three-signal picture.

Checklist

  • ✅ Some telemetry can be collected with reasonable effort (Zipkin + Prometheus scrape)
  • ✅ Tracing is documented (TracingService CRD reference exists)
  • ❌ No OTLP push for any signal
  • ❌ Native driver: opentelemetry is non-functional in the released version
  • ❌ Metrics require pull-based scraping, not OTLP push
  • ❌ No log export at all
1 Semantic Conventions Deprecated HTTP attributes throughout; Envoy-native naming not aligned with OTel semconv ▶

What we observed

All HTTP span attributes use the deprecated pre-1.20 OTel HTTP semantic conventions. Observed attributes include http.method (deprecated → http.request.method), http.status_code (deprecated → http.response.status_code), http.url (deprecated → url.full), http.protocol (deprecated → network.protocol.name/network.protocol.version), and net.host.ip (deprecated → network.local.address). Additionally, numerous Envoy-native attributes with no OTel equivalent are present: downstream_cluster, upstream_cluster, upstream_cluster.name, upstream_address, response_flags, node_id, guid:x-request-id, and component (old OpenTracing convention). All 413 metrics use Envoy-native naming (envoy_ prefix) rather than OTel dotted namespace conventions. No schemaUrl is set natively on any signal.

What's working well

  • HTTP attributes are semantically meaningful — method, status code, and URL are all present with correct values, just under deprecated key names.
  • Envoy-native attributes like upstream_cluster, response_flags, and response_size provide rich operational context beyond what OTel semconv requires.

Opportunities

Migrate to current HTTP semconv — Updating Envoy's tracing configuration to emit http.request.method, http.response.status_code, url.full, network.protocol.name, and network.local.address would align with OTel HTTP semconv v1.20+ and make spans compatible with modern observability tools without manual attribute mapping.
Set span status for error responses — Currently all spans have status: {} (UNSET) even for 4xx/5xx responses. Setting STATUS_ERROR when http.status_code ≥ 400 would enable error-rate alerting directly on span status, a core OTel capability.
Add schemaUrl to resource spans — Emitting a schema URL (e.g., https://opentelemetry.io/schemas/1.26.0) on trace resourceSpans would allow downstream tools to correctly interpret and migrate attribute names across semconv versions.
1 Resource Attributes & Configuration Only service.name emitted natively; inconsistent across signals; no OTEL_* env var support ▶

What we observed

Emissary natively emits only service.name: "ambassador-emissary" on trace spans — derived from the Envoy cluster name and not user-configurable via the TracingService CRD. No service.version, service.namespace, service.instance.id, telemetry.sdk.name, or telemetry.sdk.version are present. The Prometheus-scraped metrics path produces a different service.name: "emissary" (synthesized by the OTel Collector from the scrape target address), making cross-signal correlation by service identity impossible without pipeline-level transforms. No OTEL_* environment variables are used; Emissary uses Envoy's built-in tracing subsystem which is configured via CRD, not OTel SDK conventions.

What's working well

  • service.name is present on trace spans, providing a baseline service identity for trace-based queries.
  • After collector enrichment via the k8sattributes processor, spans gain the full Kubernetes identity set (k8s.deployment.name, k8s.namespace.name, k8s.pod.name, etc.).

Opportunities

Standardize service.name across signals — The mismatch between "ambassador-emissary" (traces) and "emissary" (metrics) prevents cross-signal correlation. Making the service name consistent and user-configurable (e.g., via a Helm value or TracingService field) would immediately enable trace-metric correlation.
Emit service.version — Including the Emissary version (e.g., 3.12.2) as a resource attribute on all signals would enable version-based filtering and canary analysis in observability tools.
Support OTEL_RESOURCE_ATTRIBUTES — Even without a full OTel SDK, exposing a mechanism (Helm value or env var passthrough) for operators to inject custom resource attributes would give users control over their service identity without requiring source code changes.
2 Trace Modeling & Context Propagation W3C Trace Context propagation works correctly; coherent end-to-end traces; span kinds incorrect ▶

What we observed

Each HTTP request through Emissary produces two Envoy spans: an ingress span (name: "localhost:18080", kind: CLIENT) and an egress span (name: "router otel_eval_backend_demo_3000 egress", kind: CLIENT). The parent-child relationship is correct. W3C Trace Context propagation is confirmed working — when a request arrives with a traceparent header, the downstream backend spans appear with the injected trace ID and parent span ID. However, when an inbound traceparent is present, Emissary does not create a child span of the incoming trace — the Emissary proxy hop becomes invisible in the distributed trace. Both ingress and egress spans use CLIENT kind; the ingress span should be SERVER per OTel semantics. Span status is never set to ERROR even for 4xx/5xx responses.

What's working well

  • W3C Trace Context (traceparent) is correctly read and forwarded to downstream services, enabling distributed tracing across service boundaries.
  • Parent-child span relationships within Emissary (ingress → egress) are correct and produce coherent traces.
  • End-to-end traces spanning Emissary ingress → egress → backend server → backend middleware are visible in a single trace ID when no inbound trace context is injected.

Opportunities

Correct span kind for ingress span — The ingress span (receiving inbound HTTP traffic) currently uses CLIENT kind. Changing it to SERVER would correctly represent Emissary's role as the server-side of the inbound connection, aligning with OTel span kind semantics. This is an Envoy-level constraint worth tracking upstream.
Create a child span when inbound traceparent is present — Currently the Emissary proxy hop is invisible when trace context is injected by the client. Creating a child span of the incoming trace would make Emissary's processing visible in distributed traces from external clients.
Use logical operation names for spans — Replacing "localhost:18080" and "router otel_eval_backend_demo_3000 egress" with names like "ingress" and "upstream: backend-demo" would make traces immediately readable to operators unfamiliar with Envoy internals.
1 Multi-Signal Observability Traces and metrics available but via non-OTLP paths; no logs; no cross-signal correlation ▶

What we observed

Two signals are available: traces (Zipkin HTTP push) and metrics (Prometheus scrape). Logs are stdout plain text only with no collection path. The two active signals use different collection models — push (Zipkin) vs. pull (Prometheus) — and different protocols, requiring separate collector pipeline configurations. Cross-signal correlation is not possible: trace spans carry no log context (no OTLP logs exist), metrics use service.name: "emissary" while traces use service.name: "ambassador-emissary", and no Prometheus exemplars linking metric data points to trace IDs are present.

What's working well

  • Two signals (traces and metrics) are available and provide meaningful operational coverage for request traffic monitoring.
  • Metrics are rich — 413 metrics covering connections, request rates, latency histograms (p50/p75/p90/p95/p99), circuit breaker states, HTTP/2 stream counts, and more.

Opportunities

Unify signals under OTLP — Once the driver: opentelemetry bug is fixed (traces) and OTLP metrics push is added, all signals could flow through a single OTLP pipeline. This would simplify collector configuration and enable consistent resource attribute propagation across signals.
Add Prometheus exemplars — Envoy supports exemplar emission on histograms. Enabling exemplars that carry trace_id and span_id would link request latency data points directly to the corresponding traces, enabling trace-metric correlation without requiring a consistent service.name.
Structured log export — Adding OTLP log export (or a documented collector-based log pipeline) would complete the three-signal picture and enable log-trace correlation via trace context injection into log records.
1 Audience & Signal Quality Span names are Envoy-internal; 413 unfiltered metrics; requires deep Envoy knowledge to interpret ▶

What we observed

Emissary span names are Envoy-internal identifiers: "localhost:18080" encodes the local port (not a logical operation), and "router otel_eval_backend_demo_3000 egress" encodes the Envoy cluster name derived from the Kubernetes service name with underscores replacing dots. An operator must know Envoy cluster naming conventions to interpret these. Metrics export all 413 Envoy statistics as-is, including zero-valued counters for edge-case error conditions (e.g., envoy_cluster_http2_inbound_priority_frames_flood, envoy_cluster_http2_outbound_control_flood). There is no default filtering or grouping. Tracing is opt-in via the TracingService CRD — an operator deploying with default settings receives no traces at all.

What's working well

  • Span attributes contain sufficient information to reconstruct full request details — method, URL, status code, upstream cluster, response flags — for operators familiar with Envoy.
  • Metrics cover the full Envoy operational surface comprehensively, including latency histograms with multiple percentile buckets, connection pool states, and circuit breaker metrics.

Opportunities

Use logical span names — Replacing "localhost:18080" with a name like "ingress" or "HTTP {method} {route}", and replacing the cluster-name-based egress span name with "upstream: {service-name}", would make traces immediately readable to operators without Envoy expertise.
Provide a curated default metric set — Offering a Helm value or documentation that highlights the 10–20 most operationally relevant metrics (request rate, error rate, latency p99, active connections) would significantly reduce noise for operators who don't need full Envoy internals visibility.
Enable tracing by default with sensible sampling — Making tracing opt-out (with a low default sampling rate) rather than opt-in would give operators immediate trace visibility without requiring CRD knowledge, dramatically improving out-of-the-box usability.
1 Stability & Change Management TracingService CRD documented; OTel driver broken in release; no telemetry schema documentation ▶

What we observed

The Emissary documentation describes the TracingService CRD and its driver options (including opentelemetry). However, the driver: opentelemetry option is documented as available but silently broken in v3.12.2 due to a code bug in python/ambassador/envoy/v3/v3tracing.py (wrong Envoy tracer extension name: "envoy.opentelemetry" instead of "envoy.tracers.opentelemetry"). There is no documentation of emitted span names, span attributes, or resource attributes. There is no documentation of Prometheus metric semantics. No schema URL is set natively on any signal. No telemetry-specific changelog entries exist. The Zipkin workaround requires community knowledge (GitHub issues, source code inspection) to discover.

What's working well

  • The TracingService CRD is documented with driver options, providing a discoverable configuration surface for tracing integration.
  • The CRD-based configuration approach provides a stable Kubernetes-native API surface that is versioned and schema-validated.

Opportunities

Document emitted telemetry as a contract — Adding a "Telemetry Reference" page documenting the span names, span attributes, resource attributes, and key Prometheus metrics that Emissary emits would give operators a stable contract to build dashboards and alerts against.
Add telemetry-specific changelog entries — Calling out changes to span names, metric names, or attribute sets in the CHANGELOG would help operators know when to update their dashboards and alert rules after upgrades.
Add a regression test for the OTel driver — The broken driver: opentelemetry could have been caught with an integration test that verifies Envoy establishes a connection to the OTel collector endpoint when configured. Adding such a test would prevent silent regressions of the advertised native OTel integration.

Checklist

  • ✅ TracingService CRD is documented with driver options
  • ❌ driver: opentelemetry is documented but broken in the released version
  • ❌ No documentation of emitted span names, attributes, or resource attributes
  • ❌ No documentation of Prometheus metric semantics
  • ❌ No schemaUrl set natively on any signal
  • ❌ No explicit stability guarantees for telemetry output
  • ❌ No telemetry-specific changelog entries
Key Takeaways

🟢 Strengths

  • W3C Trace Context propagation — When inbound requests carry a traceparent header, Emissary correctly reads and forwards the trace context to downstream services, enabling distributed tracing across service boundaries.
  • Rich Envoy metrics coverage — 413 metrics covering the full Envoy operational surface: upstream/downstream connections, request rates, latency histograms (p50–p99), circuit breaker states, and HTTP/2 stream counts.
  • Coherent end-to-end traces — Without an inbound trace context, Emissary produces coherent traces spanning the full request path: ingress span → egress span → backend server span → backend middleware spans, all in a single trace ID.

🟡 Priority Improvements

  • Fix the driver: opentelemetry bug — A one-line fix in v3tracing.py ("envoy.opentelemetry" → "envoy.tracers.opentelemetry") would enable native OTLP trace export and unlock significant maturity improvements.
  • Update HTTP semconv to current stable — Replace deprecated attributes (http.method → http.request.method, http.status_code → http.response.status_code, etc.) and set span status to ERROR for 4xx/5xx responses.
  • Standardize service.name across signals — Align "ambassador-emissary" (traces) and "emissary" (metrics) to the same value, and add service.version to enable cross-signal correlation and version-based analysis.
  • Document emitted telemetry — Add a Telemetry Reference page covering span names, attributes, and key metrics to give operators a stable contract for dashboards and alerts.

🔵 Notable Observations

  • Silent failure of the OTel driver — Configuring driver: opentelemetry produces zero errors; Envoy simply ignores the unknown tracer name and proceeds without tracing. Users who configure this will believe tracing is working when it is not — a significant trust and debuggability issue.
  • Proxy hop invisible with injected trace context — When a request arrives with a traceparent header, Emissary does not create a child span of the incoming trace. The backend spans become direct children of the externally-injected parent, making the Emissary proxy hop invisible in distributed traces from external clients.
  • service.name inconsistency is collector-introduced — The service.name: "emissary" on metrics is synthesized by the OTel Collector's Prometheus receiver from the scrape target address. Neither value is user-configurable through Emissary's configuration surface, making this a systemic issue rather than a simple misconfiguration.
  • Span kind CLIENT for ingress — Both the ingress span (receiving inbound traffic) and the egress span (sending to upstream) are modeled as CLIENT spans. This is an Envoy limitation that Emissary inherits and should be tracked as an upstream issue.
