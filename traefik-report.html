2 Integration Surface OTLP is the primary path for all three signals; Prometheus retained as secondary ▶

What we observed

Traefik v3 pushes traces, metrics, and logs directly to an OTel Collector via OTLP gRPC without any sidecar or adapter. Configuration is done through Traefik-specific Helm values (tracing.otlp, metrics.otlp, logs.*.otlp) pointing to otel-collector-opentelemetry-collector.opentelemetry.svc.cluster.local:4317. A Prometheus scrape endpoint remains available at :9100/metrics as a secondary option. OTLP log export currently requires experimental.otlpLogs: true, while traces and metrics are fully stable.

What's working well

  • OTLP gRPC export confirmed working for all three signals — no glue code or sidecars required.
  • Prometheus endpoint is clearly positioned as a secondary option, not the default, reflecting intentional OTLP-first design.
  • Documentation positions OpenTelemetry as the recommended observability path for Traefik v3 with dedicated pages for each signal.
  • Integration works cleanly with a standard OTel Collector pipeline out of the box.

Opportunities

Adopt OTEL_* environment variables — Traefik currently requires project-specific flags (--tracing.otlp.grpc.endpoint, --metrics.otlp.grpc.endpoint, --log.otlp.grpc.endpoint). Supporting OTEL_EXPORTER_OTLP_ENDPOINT alongside these flags would allow operators to configure Traefik's telemetry using the same mechanism they use for every other OTel-instrumented service in their fleet.
Graduate OTLP logs from experimental — The OTLP log export feature works reliably in v3.6.15. Removing the experimental.otlpLogs: true requirement would make logs a first-class signal and complete the integration surface, enabling operators to adopt a fully stable three-signal pipeline without experimental flags.

Checklist

  • ✅ OTLP export supported for traces, metrics, and logs
  • ✅ Users can connect to an existing OTel Collector without adapters or sidecars
  • ✅ Prometheus scraping retained as a secondary/optional path
  • ✅ Documentation positions OTLP as the recommended path
  • ✅ Integration works cleanly with a standard OTel Collector pipeline
  • ⚠️ Configuration uses project-specific flags, not OTEL_* environment variables
  • ⚠️ OTLP log export requires experimental.otlpLogs: true feature flag
2 Semantic Conventions Current OTel HTTP conventions in traces; traefik_* metrics and access log attributes diverge ▶

What we observed

Traefik's entry-point spans use current, stable OTel HTTP semantic conventions — attributes like http.request.method, http.response.status_code, url.path, url.scheme, client.address, network.peer.address, and user_agent.original are all present and correctly scoped. No deprecated attributes (http.method, http.status_code, http.target) were found in Traefik's own spans. Schema URL https://opentelemetry.io/schemas/1.40.0 is set on trace and log data. However, the traefik_* metric family uses non-OTel attribute names (code, method, entrypoint) and an older schema URL (1.18.0). Access log attributes use PascalCase proprietary names (RequestMethod, DownstreamStatus, RouterName) rather than OTel conventions.

What's working well

  • Trace spans are a strong example of OTel HTTP semantic convention alignment — attributes like http.request.method, url.path, client.address, and network.protocol.version are all current-spec compliant.
  • Traefik emits dual metric sets: OTel semconv metrics (http.server.request.duration, http.client.request.duration) alongside the proprietary traefik_* family — showing active alignment intent.
  • Schema URL https://opentelemetry.io/schemas/1.40.0 is set on trace data, indicating deliberate versioned alignment with the OTel specification.
  • The entry_point custom attribute is operationally meaningful and documented in Traefik's configuration reference.

Opportunities

Align traefik_* metric attributes with OTel conventions — The proprietary metric family uses code, method, entrypoint, router, and service as attribute names. Mapping these to http.response.status_code, http.request.method, and traefik.entrypoint (as a namespaced custom attribute) would make the traefik_* metrics compatible with generic OTel tooling and dashboards without sacrificing backward compatibility.
Align access log attributes with OTel log semantic conventions — Access log records currently use PascalCase proprietary names (RequestMethod, DownstreamStatus, RouterName). Adopting OTel log attribute names or explicitly documenting the proprietary schema as a semantic extension with a mapping table would improve interoperability with generic log analysis tools.
Set instrumentation scope version — The github.com/traefik/traefik instrumentation scope currently reports vunknown as its version. Setting this to the actual Traefik release version would allow telemetry backends to correlate scope metadata with the release that produced the data.

Checklist

  • ✅ Current OTel HTTP semantic conventions used in trace spans (no deprecated http.method, http.status_code)
  • ✅ OTel semconv metrics (http.server.request.duration, http.client.request.duration) emitted alongside proprietary metrics
  • ✅ Schema URL set on trace and log data (https://opentelemetry.io/schemas/1.40.0)
  • ⚠️ traefik_* metrics use non-OTel attribute names (code, method, entrypoint)
  • ⚠️ Access log attributes use PascalCase proprietary names, not OTel conventions
  • ⚠️ Instrumentation scope version not set (vunknown)
  • ⚠️ Access log severity mapping has a bug ("panic" in body, info at OTLP field level)
2 Resource Attributes & Configuration Rich native identity across all signals; configuration uses Traefik-specific flags, not OTEL_* env vars ▶

What we observed

Traefik natively emits a rich set of resource attributes via its embedded OTel Go SDK: service.name=traefik, service.version=3.6.15, telemetry.sdk.name=opentelemetry, telemetry.sdk.language=go, telemetry.sdk.version=1.43.0, process.executable.name, process.executable.path, process.pid, process.owner, process.runtime.*, os.type=linux, os.description, and host.name. A native Kubernetes resource detector (added in PR #11906) additionally sets k8s.pod.uid, k8s.pod.name, and k8s.namespace.name. All attributes are consistent across traces, metrics, and logs. However, Traefik does not respect OTEL_EXPORTER_OTLP_ENDPOINT, OTEL_SERVICE_NAME, or OTEL_RESOURCE_ATTRIBUTES — all OTel configuration must go through Traefik-specific CLI flags.

What's working well

  • service.name and service.version are set natively and consistently across all three signals — no collector enrichment needed for service identity.
  • Rich process, OS, and host attributes (process.executable.path, process.runtime.version, os.description) are emitted natively, providing deep operational context without any pipeline configuration.
  • Native Kubernetes pod identity (k8s.pod.uid, k8s.pod.name, k8s.namespace.name) is unusual and impressive for an ingress controller — most projects rely entirely on collector enrichment for k8s attributes.
  • Identity is fully consistent across all three signals — service.name, service.version, and telemetry.sdk.* attributes match exactly across traces, metrics, and logs.

Opportunities

Support OTEL_* environment variables — Adding support for OTEL_EXPORTER_OTLP_ENDPOINT, OTEL_SERVICE_NAME, and OTEL_RESOURCE_ATTRIBUTES alongside Traefik's own flags would allow operators to configure Traefik's telemetry using the same mechanism they use for every other OTel-instrumented service, reducing cognitive overhead in multi-service deployments.
Add service.instance.id — Currently there is no per-instance differentiator beyond host.name (the pod hostname). Setting service.instance.id to a stable per-pod identifier (e.g., the pod UID) would allow backends to distinguish metrics and traces from individual Traefik replicas in a multi-replica deployment.

Checklist

  • ✅ service.name and service.version set natively and consistently across all signals
  • ✅ Identity is stable across restarts (tied to the binary version, not runtime state)
  • ✅ Resource attributes placed correctly (not on spans)
  • ✅ Rich process, OS, and host attributes emitted natively
  • ✅ Kubernetes pod identity emitted natively (k8s.pod.uid, k8s.pod.name, k8s.namespace.name)
  • ⚠️ service.instance.id not set (no per-instance differentiation beyond host.name)
  • ❌ OTEL_EXPORTER_OTLP_ENDPOINT, OTEL_SERVICE_NAME, OTEL_RESOURCE_ATTRIBUTES not respected — configuration uses Traefik-specific flags only
2 Trace Modeling & Context Propagation Coherent SERVER→CLIENT trace structure; W3C Trace Context propagated end-to-end ▶

What we observed

Traefik produces a clean two-span structure per routed HTTP request: a root SERVER span (named by HTTP method, e.g., GET) representing the entry-point, and a child CLIENT span (ReverseProxy) representing the upstream proxy call. W3C Trace Context propagation is confirmed working — Traefik continues incoming traces via traceparent headers and injects traceparent into upstream requests when none is present. The full distributed trace chain GET (traefik, SERVER) → ReverseProxy (traefik, CLIENT) → GET / (backend, SERVER) was confirmed end-to-end. Error spans correctly set status.code=2 (ERROR). Configurable verbosity (minimal vs. detailed) allows operators to add middleware spans when needed.

What's working well

  • Span kinds are correct: SERVER for entry-point spans, CLIENT for upstream proxy calls — following OTel semantic conventions precisely.
  • Span names follow OTel HTTP semantic conventions for server spans (HTTP method-based naming), confirmed as an intentional decision via PR #11673 ("[tracing] Follow OTel semantic conventions for root span naming").
  • W3C Trace Context propagation works end-to-end — Traefik correctly continues incoming traces and starts new traces when no context is present.
  • The entry_point custom attribute (e.g., web, traefik) is a useful domain-specific addition that helps operators identify which Traefik entrypoint handled a request in multi-entrypoint deployments.
  • Configurable trace verbosity (addInternals: true, detailed mode) gives operators control over signal volume without sacrificing default usability.

Opportunities

Document trace modeling decisions — Traefik's trace structure (SERVER→CLIENT span hierarchy, method-based naming, entry_point attribute) is well-designed but not explicitly documented as an architectural decision. Adding a "How Traefik models traces" section to the documentation would help operators understand what to expect and set up dashboards and alerts with confidence.
Document trace behavior for middleware chains and retries — The trace structure for middleware chains, retry logic, and circuit-breaker scenarios is not explicitly documented. Adding examples of these trace topologies would help operators understand Traefik's observability behavior in complex routing scenarios.

Checklist

  • ✅ W3C Trace Context (traceparent/tracestate) propagated to upstream services
  • ✅ SERVER spans for entry points, CLIENT spans for upstream proxy calls
  • ✅ Correct parent-child relationships between entry-point and proxy spans
  • ✅ Span names follow OTel HTTP semantic conventions (method-based)
  • ✅ Distributed traces confirmed end-to-end (Traefik → backend)
  • ✅ Error status correctly set on failed spans (status.code=2)
  • ⚠️ Trace behavior for async/background work (middleware chains, retries) not explicitly documented
2 Multi-Signal Observability All three signals via OTLP; access logs carry trace/span IDs for native log-to-trace correlation ▶

What we observed

Traefik v3 emits all three signals via OTLP gRPC: traces (stable), metrics (stable, every 10s), and logs (experimental). Access log records include traceId and spanId at the OTLP record level, confirmed in the telemetry data — enabling native log-to-trace correlation in backends like Grafana without additional processing. All signals share the same service.name=traefik and service.version=3.6.15 resource identity, enabling cross-signal filtering. However, the traefik_* metric attribute names (code, method) differ from trace attribute names (http.response.status_code, http.request.method), requiring mapping for cross-signal correlation on those dimensions. General daemon logs do not include trace context — only access logs do.

What's working well

  • All three signals are first-class OTLP outputs — traces, metrics, and logs all flow through the same OTLP gRPC pipeline to the collector.
  • Access logs include traceId and spanId at the OTLP record level, enabling native log-to-trace linking in observability backends without any additional processing or log parsing.
  • Consistent service.name and service.version across all three signals enables cross-signal filtering and correlation by service identity.
  • Metrics and traces complement each other well: http.server.request.duration provides aggregate latency and error rate, while traces provide per-request detail.

Opportunities

Align metric attribute names with trace attribute names — The traefik_* metric family uses code and method as attribute names, while traces use http.response.status_code and http.request.method. Aligning these (or documenting the mapping) would make it easier to correlate specific metric dimensions with the corresponding trace attributes in dashboards and alert queries.
Add trace context to general daemon logs — Currently, only access logs include traceId and spanId. Adding trace context to general daemon logs (configuration events, error logs) where a request context is available would enable operators to correlate configuration-level events with the traces that triggered them.

Checklist

  • ✅ All three signals emitted via OTLP
  • ✅ Access logs include traceId and spanId at the OTLP record level — native log-to-trace correlation
  • ✅ Consistent service.name and service.version across all signals
  • ✅ Metrics and traces complement each other (aggregate counts/latencies vs. per-request detail)
  • ⚠️ OTLP log export is experimental — not yet a stable first-class signal
  • ⚠️ General daemon logs do not include trace context (only access logs do)
  • ⚠️ traefik_* metric attribute names (code, method) differ from trace attribute names — cross-signal correlation requires mapping
2 Audience & Signal Quality Operator-oriented span names and attributes; access log naming and severity issues reduce out-of-the-box usability ▶

What we observed

Traefik's telemetry is clearly designed with operators in mind. Entry-point spans use HTTP method names (GET, POST) as span names — immediately understandable — and include all information needed to understand a request: method, path, status code, client address, user agent, and the upstream backend. The entry_point attribute identifies which Traefik entrypoint handled the request. Configurable trace verbosity (minimal vs. detailed) gives operators control over signal volume. However, access log attributes use PascalCase proprietary names (RequestMethod, DownstreamStatus, RouterName) rather than OTel conventions, and there is a known severity mapping bug where access log records report severityText="info" at the OTLP level but contain "level":"panic" in the log body.

What's working well

  • Span names follow OTel HTTP semantic conventions (HTTP method-based for SERVER spans, ReverseProxy for CLIENT spans) — confirmed as an intentional quality improvement via PR #11673.
  • Span attributes provide complete operational context: entry_point, http.request.method, url.path, http.response.status_code, client.address, user_agent.original, and upstream server.address are all present by default.
  • Configurable trace verbosity (minimal default, detailed for middleware spans, addInternals: true for health check traces) is well-designed — operators choose their verbosity level without sacrificing default usability.
  • Metrics cover the full request lifecycle at three levels of granularity: entry point, router, and service — giving operators multiple aggregation dimensions.

Opportunities

Fix access log severity mapping — Access log records currently report severityText="info" at the OTLP record level but contain "level":"panic" in the log body. This is a known bug in the experimental OTLP log implementation. Fixing the severity mapping would ensure that log-based alerting on severity level works correctly out of the box.
Align access log attribute names with OTel conventions — Replacing PascalCase proprietary names (RequestMethod, DownstreamStatus, RouterName) with OTel log semantic convention names (or explicitly documenting the mapping) would make access logs usable in generic OTel log analysis tools without project-specific knowledge.
Consolidate duplicate trace/span ID fields in access logs — Access logs currently include TraceId, SpanId, trace_id, and span_id as log attributes in addition to the OTLP record-level fields. Consolidating to a single canonical set would reduce confusion and log payload size.

Checklist

  • ✅ Span names follow OTel conventions (HTTP method-based)
  • ✅ Span attributes provide complete operational context without internal implementation details
  • ✅ Configurable trace verbosity (minimal/detailed) with sensible defaults
  • ✅ Metrics cover operationally relevant dimensions (entry point, router, service, latency, bytes)
  • ✅ Access logs include trace context for correlation
  • ⚠️ Access log attribute names use PascalCase (RequestMethod, DownstreamStatus) rather than OTel conventions
  • ⚠️ Access log severity bug: severityText="info" at OTLP level but body contains "level":"panic"
  • ⚠️ General logs lack severityText at the OTLP record level (only severityNumber set)
  • ⚠️ Duplicate trace/span ID fields in access logs (TraceId, SpanId, trace_id, span_id)
1 Stability & Change Management OTel changes appear in changelogs; no formal telemetry stability contract or deprecation policy ▶

What we observed

Traefik's changelog includes OTel-related entries tagged with [otel], [tracing], [metrics], [logs], and [accesslogs]. Recent examples include: adding OTel-conformant trace context attributes to access logs (#12801), updating OTel to v1.38.0 and semantic conventions to v1.37.0 (#12099), following OTel semantic conventions for root span naming (#11673), renaming traefik_tls_certs_not_after_milliseconds to traefik_tls_certs_not_after_seconds (#12141), and adding k8s resource attributes natively (#11906). The project is aware that OTel changes have impact and labels experimental features appropriately. However, there is no formal telemetry stability contract, no explicit "breaking change" flag for telemetry changes, and no migration guidance when metric names or span attributes change.

What's working well

  • OTel-related changes are consistently tagged in the changelog ([otel], [tracing], [metrics]), making it possible for users to track telemetry changes across releases.
  • Experimental features are explicitly labeled — experimental.otlpLogs: true is a clear signal to users that this feature is not yet stable, demonstrating stability awareness.
  • Schema URL is set on trace and log data (https://opentelemetry.io/schemas/1.40.0), providing a versioned reference for the semantic conventions in use.
  • The project actively tracks OTel semantic convention updates — the changelog shows regular bumps of go.opentelemetry.io/otel and semantic convention versions.

Opportunities

Establish a telemetry stability contract — Documenting which telemetry signals, metric names, span attributes, and log attributes are considered stable (and which are experimental) would give operators confidence when building dashboards and alerts on Traefik's telemetry. A dedicated "Telemetry Reference" page listing stable vs. experimental telemetry would be a significant improvement.
Flag breaking telemetry changes explicitly with migration guidance — Breaking changes like the metric rename from traefik_tls_certs_not_after_milliseconds to traefik_tls_certs_not_after_seconds affect users with existing dashboards and alerts. Adding explicit "BREAKING" flags to telemetry changes in release notes, along with a migration note (e.g., "update your Prometheus alert rule from X to Y"), would significantly reduce operator friction during upgrades.
Define a review process for telemetry changes — Establishing a lightweight process (e.g., a checklist in the PR template for telemetry-affecting changes) would ensure that breaking telemetry changes are identified and communicated consistently before they ship.

Checklist

  • ✅ OTel-related changes appear in changelogs with relevant tags
  • ✅ Experimental features explicitly labeled (experimental.otlpLogs)
  • ✅ Schema URL set on trace and log data
  • ⚠️ No explicit telemetry stability contract or documentation
  • ⚠️ Breaking telemetry changes (e.g., metric renames) listed in changelog but without migration guidance
  • ⚠️ No distinction between stable and experimental telemetry beyond the experimental.otlpLogs flag
  • ❌ No defined review process for telemetry changes
  • ❌ No deprecation timelines or migration paths for telemetry changes
Key Takeaways

🟢 Strengths

  • Excellent trace semantic convention alignment — Entry-point spans use current, stable OTel HTTP conventions with no deprecated attributes and schema URL set.
  • Rich native resource attributes — service.name, service.version, process/OS/host, and native Kubernetes pod identity (k8s.pod.uid, k8s.pod.name) emitted consistently across all three signals.
  • Full distributed tracing with W3C context propagation — The complete trace chain (Traefik entry-point → proxy → backend) works out of the box with correct span kinds and parent-child relationships.
  • Native log-to-trace correlation — Access logs include traceId and spanId at the OTLP record level, enabling native correlation in backends like Grafana without additional processing.
  • Dual metric strategy — Both OTel semconv metrics (http.server.request.duration) and proprietary traefik_* metrics are emitted, showing active alignment intent while maintaining backward compatibility.

🟡 Priority Improvements

  • Adopt OTEL_* environment variable support — Respecting OTEL_EXPORTER_OTLP_ENDPOINT, OTEL_SERVICE_NAME, and OTEL_RESOURCE_ATTRIBUTES would enable standard OTel SDK configuration patterns across the ecosystem.
  • Fix access log severity mapping — The "level":"panic" bug in access log bodies is a known issue in the experimental OTLP log implementation that affects log-based alerting.
  • Graduate OTLP logs from experimental — OTLP log export works reliably in v3.6.15; removing the experimental.otlpLogs: true requirement would make logs a stable first-class signal.
  • Establish a telemetry stability contract — Documenting stable vs. experimental telemetry and flagging breaking changes with migration guidance would significantly reduce operator friction during upgrades.
  • Align access log attribute names with OTel conventions — Replacing PascalCase proprietary names with OTel semantic convention names would improve interoperability with generic log analysis tools.

🔵 Notable Observations

  • Native Kubernetes resource detector — Traefik includes a native k8s resource detector (PR #11906) that sets k8s.pod.uid, k8s.pod.name, and k8s.namespace.name without collector enrichment — unusual and impressive for an ingress controller.
  • Dual schema URL versions — The traefik_* metric scope uses schema URL https://opentelemetry.io/schemas/1.18.0 while traces and OTel semconv metrics use 1.40.0, suggesting the proprietary metrics were not updated when the schema was bumped.
  • Instrumentation scope version not set — The github.com/traefik/traefik scope reports vunknown, making it impossible to correlate scope metadata with a specific Traefik version (though service.version in the resource serves this purpose).
  • process.command_args exposes OTel endpoint configuration — The full Traefik command line (including OTLP endpoint addresses) appears in the resource attributes, which may expose internal configuration details in shared observability backends.
  • Active OTel maintenance — The changelog shows regular bumps of go.opentelemetry.io/otel and semantic convention versions, indicating Traefik actively tracks and adopts OTel specification updates.
