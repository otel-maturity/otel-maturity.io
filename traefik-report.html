Dimension details
2 Integration Surface OTLP is a first-class, well-documented path; Prometheus scrape remains co-equal ▶

What we observed

Traefik v3 supports OTLP gRPC and OTLP HTTP export natively for all three signals — traces via tracing.otlp.grpc, metrics via metrics.otlp.grpc, and logs via experimental.otlpLogs: true. In our evaluation, Traefik connected directly to an OTel Collector on port 4317 with zero custom adapters or sidecars. Prometheus scrape (metrics.prometheus) remains a co-equal first-class option alongside OTLP push, and OTLP log export still requires the experimental feature gate.

What's working well

  • OTLP export is supported natively for all three signals using the OTel Go SDK — no wrappers, bridges, or sidecars required.
  • Configuration is consistent across signals — all use the *.otlp.grpc.* pattern — and the Helm chart exposes tracing.otlp and metrics.otlp as first-class value keys.

Opportunities

Stabilize OTLP log export — Graduating experimental.otlpLogs: true to a stable, default-on feature would complete the three-signal story and remove the friction of opting into log export.
Clarify Prometheus deprecation path — Adding documentation guidance on when to prefer OTLP push over Prometheus scrape — and a roadmap for making OTLP the primary metrics path — would reduce ambiguity for new operators.

Checklist

  • ✓ OTLP export supported for all three signals via OTel Go SDK
  • ✓ Connects directly to an existing OTel Collector without adapters or sidecars
  • ✗ Prometheus scrape remains co-equal with OTLP push — not clearly secondary or deprecated
  • ✗ OTLP log export requires experimental.otlpLogs: true — not yet a stable default
2 Semantic Conventions Trace spans use current semconv; Prometheus metrics use legacy labels; access log attributes are Traefik-native CamelCase ▶

What we observed

Traefik v3 trace spans use the latest stable OTel HTTP semantic conventions from semconv/v1.26.0. Attributes like http.request.method, http.response.status_code, url.path, url.full, url.scheme, server.address, client.address, and network.protocol.version are all present — no deprecated fields (http.method, http.status_code, http.target) were observed. OTLP metrics use OTel semconv names (http.server.request.duration, http.client.request.duration). However, Prometheus metrics use Traefik-native labels (code, method, protocol) and access logs use a Traefik-native CamelCase schema (RequestMethod, DownstreamStatus, etc.).

What's working well

  • All trace span HTTP attributes use current stable semconv — no deprecated attributes (http.method, net.host.*, net.peer.*) were observed. Span kinds are correct: SERVER (kind=2) for entry-point spans, CLIENT (kind=3) for ReverseProxy spans.
  • OTLP push metrics use OTel semconv names and attribute keys: http.server.request.duration with http.request.method, http.response.status_code, url.scheme, and server.address — directly compatible with OTel-based dashboards.

Opportunities

Align Prometheus metric label names with OTel semconv — The traefik_entrypoint_requests_total metric uses code, method, and protocol labels. Updating these to http.response.status_code, http.request.method, and network.protocol.name would enable natural pivot from Prometheus dashboards to traces.
Adopt OTel semconv for access log attributes — Access log attributes currently use Traefik-native CamelCase (RequestMethod, DownstreamStatus, RouterName). Mapping these to OTel semconv equivalents (http.request.method, http.response.status_code) would make logs interpretable by generic OTel tooling without project-specific knowledge.
2 Resource Attributes & Configuration Rich native resource attributes; OTEL_RESOURCE_ATTRIBUTES documented and respected; service.instance.id absent natively ▶

What we observed

Traefik sets a rich set of resource attributes natively via the OTel Go SDK: service.name=traefik, service.version=3.7.0, telemetry.sdk.name, telemetry.sdk.language, telemetry.sdk.version, plus process attributes (process.pid, process.executable.name, process.runtime.version), OS attributes (os.type, os.description), and Kubernetes attributes (k8s.namespace.name, k8s.pod.name, k8s.pod.uid, k8s.deployment.name) detected natively via Traefik's built-in K8s resource detector. OTEL_RESOURCE_ATTRIBUTES is documented and supported. The main gap is the absence of service.instance.id, and export endpoints use Traefik-native config paths rather than OTEL_EXPORTER_OTLP_ENDPOINT.

What's working well

  • service.name and service.version are stable and consistent across all three signals. Kubernetes resource attributes (k8s.namespace.name, k8s.pod.name, k8s.pod.uid, k8s.deployment.name) are set natively — no collector enrichment required for basic K8s identity.
  • OTEL_RESOURCE_ATTRIBUTES is explicitly documented at doc.traefik.io as a supported mechanism for adding custom resource attributes, giving operators a standard override path.

Opportunities

Add service.instance.id — The pod UID is already available as k8s.pod.uid. Mapping it to service.instance.id natively would provide the standard OTel instance identity attribute that backends use for per-instance filtering and alerting.
Support OTEL_EXPORTER_OTLP_ENDPOINT for export configuration — Currently, export endpoints are configured via Traefik-native flags (tracing.otlp.grpc.endpoint). Supporting OTEL_EXPORTER_OTLP_ENDPOINT as a standard override would align with the OTel specification and improve portability across environments.
2 Trace Modeling & Context Propagation W3C Trace Context propagation confirmed; intentional SERVER/CLIENT span kinds; span names are HTTP-method-only (not route-templated) ▶

What we observed

Traefik v3 produces a clean two-level trace structure: a SERVER span named GET (the HTTP method) for the inbound entry-point request, and a CLIENT span named ReverseProxy as a child for the outbound backend call. W3C Trace Context propagation is verified — incoming traceparent headers are correctly honored, with Traefik creating child spans that preserve the trace ID. Backend spans appear as children of Traefik's ReverseProxy span. The main limitation is that span names use only the HTTP method (GET) without a route template or http.route attribute, making it impossible to distinguish routes in a trace waterfall without inspecting url.path.

What's working well

  • W3C Trace Context (traceparent/tracestate) propagation is correctly implemented and verified end-to-end. Traefik preserves the incoming trace ID and creates child spans, enabling distributed tracing across the ingress boundary.
  • Span kinds are intentionally correct — SERVER (kind=2) for entry-point spans and CLIENT (kind=3) for ReverseProxy spans — producing a coherent trace structure: [external] → GET (SERVER) → ReverseProxy (CLIENT) → backend spans.

Opportunities

Add http.route to SERVER spans — Traefik knows the matched router name at span creation time (available as RouterName in access logs). Exposing it as http.route or a Traefik-specific attribute like traefik.router.name would transform generic GET spans into route-aware GET /api/v1/users/{id}, dramatically improving trace usability for ingress-level analysis.
Set instrumentation scope version on traces — The trace instrumentation scope (github.com/traefik/traefik) currently reports version vunknown. Setting the scope version to the Traefik release version would help backends track semconv version changes across upgrades.
2 Multi-Signal Observability All three signals flow via OTLP; access logs carry trace/span IDs; OTLP logs remain behind an experimental feature gate ▶

What we observed

All three signals are present and flowing via OTLP gRPC. Cross-signal correlation works natively: all 28 access log records observed have traceId and spanId populated in OTLP fields, enabling pivot from logs to traces. OTLP metrics share attribute keys with trace spans (http.request.method, http.response.status_code, url.scheme, server.address), enabling pivot from OTLP metrics to traces. However, Prometheus metrics use Traefik-native label names (code, method, entrypoint) that don't match trace span attributes, and OTLP log export requires experimental.otlpLogs: true.

What's working well

  • Access logs carry traceId and spanId in OTLP fields natively — cross-signal correlation from logs to traces works without any collector-side enrichment or manual stitching.
  • OTLP metrics and traces share attribute keys (http.request.method, http.response.status_code, server.address), enabling natural pivot from latency metrics to specific traces using the same dimension values.

Opportunities

Stabilize OTLP log export — Removing the experimental.otlpLogs: true gate would make all three signals available by default in OTel-native environments, completing the full correlation story without operator opt-in.
Add cross-signal navigation guidance to documentation — A documentation page explaining how to navigate between signals — e.g., "from a slow traefik_entrypoint_request_duration_seconds spike, find the trace" — would help operators take full advantage of Traefik's multi-signal observability.
1 Audience & Signal Quality Trace and metrics signals are operator-ready; access log attribute schema is Traefik-native (not OTel semconv); level:panic body quirk for all access logs ▶

What we observed

Traefik's trace and OTLP metrics signals are operationally useful — spans carry rich HTTP context and the http.server.request.duration histogram is directly compatible with OTel-aware dashboards. However, the access log signal has several quality issues: all access log records embed "level":"panic" in the JSON body regardless of HTTP status (a known serialization quirk), log attributes duplicate trace context in both CamelCase (TraceId, SpanId) and snake_case (trace_id, span_id), and the log body is a JSON string (not a structured object) — redundant with extracted attributes. SERVER span names are HTTP-method-only (GET), with no http.route attribute.

What's working well

  • The traefik_* Prometheus metrics are operationally useful for dashboards — request rates, durations, open connections, and TLS certificate expiry are all exposed. The OTLP http.server.request.duration and http.client.request.duration histograms are directly compatible with OTel-based SLO tooling.
  • Access logs provide rich per-request routing context — RouterName, ServiceName, KubernetesIngressName, RetryAttempts, Overhead — that is valuable for debugging routing issues even if the attribute names are Traefik-native.

Opportunities

Fix the level:panic access log body quirk — All access log records currently embed "level":"panic" in the JSON body regardless of HTTP status code. The OTLP severityText is correctly set to info, but the embedded body is misleading and can cause alert fatigue if log-level alerting is based on body content. Fixing this serialization quirk would improve log signal quality significantly.
Deduplicate trace context in log attributes — Access log records currently carry both TraceId/SpanId (CamelCase) and trace_id/span_id (snake_case) as duplicate log attributes, in addition to the OTLP traceId/spanId fields. Consolidating to the standard OTel snake_case attributes would reduce payload size and avoid confusion.
1 Stability & Change Management OTel changes appear in changelog; no formal telemetry stability contract; OTLP logs still experimental; no schema URL on traces ▶

What we observed

Traefik tracks OTel-related changes in its CHANGELOG with a dedicated [otel] tag — semconv version bumps, span naming changes, metric renames, and behavioral changes to span emission are all noted. Schema URL is set on OTLP metrics (https://opentelemetry.io/schemas/1.40.0) and logs, but absent on traces. The experimental.otlpLogs gate is the only formal stability marker; no equivalent stability labels exist for traces or metrics. Breaking changes (e.g., traefik_tls_certs_not_after_milliseconds renamed to traefik_tls_certs_not_after_seconds) appear in the changelog without migration guides.

What's working well

  • OTel-related changes are tracked in the CHANGELOG with a dedicated [otel] tag, making it possible to audit telemetry changes across releases. Semconv version bumps (e.g., "Update OpenTelemetry to v1.38.0 and semantic conventions to v1.37.0") are explicitly called out.
  • Schema URL https://opentelemetry.io/schemas/1.40.0 is correctly set on OTLP metrics and log exports, enabling schema-aware tooling to interpret the telemetry correctly.

Opportunities

Add schemaUrl to trace exports — OTLP metrics and logs correctly set schemaUrl=https://opentelemetry.io/schemas/1.40.0. Adding the same schema URL to trace exports would ensure consistency across all three signals and enable schema-aware backends to process traces correctly.
Publish a telemetry stability contract with migration guides — A dedicated telemetry stability page — listing which span names, attribute names, and metric names are stable vs. experimental — would help operators build reliable dashboards and alerts. Breaking changes like the traefik_tls_certs_not_after_milliseconds → traefik_tls_certs_not_after_seconds rename warrant a migration guide, not just a changelog entry.
Key takeaways

● Strengths

  • Comprehensive native OTLP support — All three signals flow via OTLP gRPC natively with no sidecars or adapters. Traefik connects directly to an OTel Collector using the OTel Go SDK.
  • Current semantic conventions on traces — Traefik v3 uses the latest stable OTel HTTP semconv (semconv/v1.26.0) on all trace spans — no deprecated attributes observed. Attributes include http.request.method, url.path, server.address, network.protocol.version, and more.
  • Rich native resource attributes including Kubernetes metadata — Traefik natively sets k8s.namespace.name, k8s.pod.name, k8s.pod.uid, k8s.deployment.name via a built-in K8s resource detector — reducing pipeline complexity and ensuring K8s identity is present even in minimal collector configurations.

● Priority improvements

  • Add http.route to SERVER spans — Exposing the matched router name as http.route would transform all-identical GET spans into route-aware spans, dramatically improving trace usability for an ingress controller.
  • Fix the level:panic access log body quirk — All access log records embed "level":"panic" in the JSON body regardless of HTTP status — a serialization quirk that causes confusion and potential alert fatigue.
  • Stabilize OTLP log export — Graduating experimental.otlpLogs: true to a stable, default-on feature would complete the three-signal story and remove operator friction.
  • Add schemaUrl to traces and set instrumentation scope version — Traces currently lack schemaUrl (present on metrics and logs) and the instrumentation scope reports version vunknown. Both are small fixes with meaningful tooling compatibility benefits.

● Notable observations

  • Dual metrics paths create semantic inconsistency — Traefik simultaneously pushes OTel-semconv metrics (http.server.request.duration with http.request.method) and exposes Prometheus metrics (traefik_entrypoint_requests_total with method) for the same underlying data. This pragmatic backward-compatibility choice creates a split observability experience.
  • Built-in K8s resource detector is a standout feature — Unlike most projects that rely entirely on the collector's k8sattributes processor, Traefik natively detects and sets Kubernetes resource attributes via a built-in K8s resource detector (PR #11906). This is a meaningful architectural advantage for minimal pipeline configurations.
  • Access log attributes are rich but non-standard — The access log schema (RouterName, ServiceName, KubernetesIngressName, Duration, Overhead, RetryAttempts) provides valuable ingress-specific context not available in traces. However, Traefik-native names mean these attributes cannot be interpreted by generic OTel tooling without project-specific knowledge.
