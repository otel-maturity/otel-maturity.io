Dimension details
1 Integration Surface OTLP supported for traces and logs (data plane); metrics are Prometheus-only with no OTLP push path ▶

What we observed

kgateway supports OTLP gRPC natively for two of its three signals. Traces from the Envoy data plane are sent via OTLP gRPC to a configurable BackendRef (Kubernetes Service), confirmed flowing to collector port 4317. Access logs are delivered via OTel ALS (OTLP gRPC) to the same endpoint. Both are configured through kgateway-specific CRDs — ListenerPolicy.spec.default.httpSettings.tracing.provider.openTelemetry.grpcService for traces and ListenerPolicy.spec.default.httpSettings.accessLog[].openTelemetry.grpcService for logs. Metrics, however, are Prometheus-only: the controller exposes /metrics on port 9092 and the Envoy proxy on port 9091, with no OTLP push path available. The official docs acknowledge this and recommend using the OTel Collector as a Prometheus scraper to bridge metrics into the OTel ecosystem.

What's working well

  • OTLP gRPC is the native transport for both traces and logs — no sidecar or adapter needed for these two signals.
  • The official docs include a dedicated "Set up the OpenTelemetry stack" guide (/docs/envoy/latest/observability/otel-stack/) covering all three signals, positioning OTel as the recommended integration path.

Opportunities

Add OTLP metrics export — Adding an OTLP push exporter to the kgateway controller would complete the integration surface and eliminate the Prometheus-scrape dependency for metrics, making all three signals native OTLP.
Simplify cross-namespace setup — When the ListenerPolicy and OTel Collector are in different namespaces, a ReferenceGrant is required. Documenting this prominently and potentially providing a helper resource would reduce integration friction for production setups.

Checklist

  • ✓ OTLP is supported for traces and logs
  • ✓ OTel Collector is the documented integration path with a dedicated official guide
  • ✗ Metrics have no OTLP push path — Prometheus scrape only
  • ✗ Configuration uses project-specific CRDs, not standard OTEL_* environment variables
0 Semantic Conventions Envoy emits deprecated HTTP attributes (http.method, http.url, http.status_code) with wrong types; no current semconv alignment ▶

What we observed

Envoy's OTel tracer emits HTTP span attributes using pre-v1.23 OpenTelemetry semantic conventions. All observed HTTP attributes are deprecated: http.method (replaced by http.request.method), http.url (replaced by url.full), and http.status_code (replaced by http.response.status_code). Additionally, http.status_code is emitted as a stringValue: "200" instead of the correct intValue: 200. Several proprietary Envoy-internal attributes are also present: upstream_cluster, node_id, response_flags, downstream_cluster, and component: "proxy". Metric names follow Prometheus underscore convention (kgateway_controller_reconcile_duration_seconds) rather than OTel dot notation. Log records have an empty body with no structured attributes.

What's working well

  • Trace-log correlation fields (traceId, spanId) are correctly placed in log records following the OTel log data model — this is a key semconv compliance win.
  • The kgateway controller's kgateway_* metric names are consistent and operationally meaningful, even if they follow Prometheus rather than OTel naming conventions.

Opportunities

Migrate to current HTTP semantic conventions — Envoy's OTel tracer currently uses deprecated HTTP attributes. Tracking Envoy's upstream semconv migration and contributing to it would bring http.request.method, url.full, and http.response.status_code (as integer) into kgateway's telemetry output.
Populate OTel access log body and attributes — Envoy OTel ALS log records currently have an empty body ({}) and no log attributes. Including HTTP method, path, status code, and latency in the log body or attributes would make OTel access logs a first-class signal for request-level analysis.
1 Resource Attributes & Configuration service.name present but inconsistent across signals; service.version absent; no OTEL_* env var support ▶

What we observed

The Envoy data plane emits service.name: kgateway-envoy.demo in traces (configurable via ListenerPolicy.tracing.serviceName) but service.name: demo-gateway.demo in logs — a different value derived from Envoy's internal cluster name format for the same pod. This inconsistency breaks cross-signal correlation by service identity. The controller does not emit service.name natively; it is assigned by the Prometheus receiver in the collector as kgateway-controller. service.version is absent from all three signals. Standard OTEL_* environment variables (OTEL_EXPORTER_OTLP_ENDPOINT, OTEL_SERVICE_NAME, OTEL_RESOURCE_ATTRIBUTES) have no effect on kgateway's telemetry behavior — configuration is exclusively through kgateway CRDs and Helm values.

What's working well

  • service.name is present and configurable in traces via ListenerPolicy.tracing.serviceName — operators have control over the trace identity.
  • telemetry.sdk.name: envoy, telemetry.sdk.language: cpp, and telemetry.sdk.version are correctly populated in trace resource attributes, identifying the underlying SDK.

Opportunities

Align service.name across traces and logs — The Envoy proxy currently emits service.name: kgateway-envoy.demo in traces but service.name: demo-gateway.demo in logs for the same process. Ensuring both signals use the same configured service name would enable reliable cross-signal correlation by service identity.
Add service.version to resource attributes — service.version is absent from all three signals. Adding the kgateway release version (e.g., v2.2.4) as a resource attribute would improve identity completeness and enable version-based filtering in observability backends.
2 Trace Modeling & Context Propagation Coherent ingress/egress span tree; W3C Trace Context propagated end-to-end with confirmed backend correlation ▶

What we observed

Each HTTP request through the Envoy gateway produces a well-structured trace tree: an ingress span (kind=SERVER) as the root, with a router kube_demo_otel-eval-backend_3000 egress span (kind=CLIENT) as its child, and the backend's GET / span correctly parented to the egress span. This three-level hierarchy accurately models the gateway's proxy role. W3C Trace Context (traceparent header) is injected by Envoy into upstream requests, and confirmed end-to-end: all spans share the same traceId (e.g., d374625072e87cb0d55ddab5c20eca14), and the backend's GET / span has parentSpanId matching the Envoy egress span's spanId. Upstream span generation requires spawnUpstreamSpan: true in ListenerPolicy — without it, only the ingress span is produced.

What's working well

  • W3C Trace Context propagation is confirmed end-to-end — the backend's spans are correctly parented to Envoy's egress span, enabling full distributed trace visibility across the gateway boundary.
  • Span kinds are correctly assigned: ingress uses SERVER kind (entry-point listener) and router ... egress uses CLIENT kind (upstream connection) — accurately modeling the proxy's dual role.

Opportunities

Enable spawnUpstreamSpan by default — Without spawnUpstreamSpan: true in ListenerPolicy, only the ingress span is generated — the proxy-to-backend connection is invisible in traces. Since kgateway exists specifically to forward requests, making upstream span generation the default would significantly improve out-of-the-box trace utility.
Set schemaUrl in OTLP trace exports — OTLP trace exports currently have an empty schemaUrl field. Setting it to the appropriate OTel semconv schema URL (e.g., https://opentelemetry.io/schemas/1.7.0 for the current attributes) would allow consumers to know which convention version is in use.
2 Multi-Signal Observability All three signals present and correlated; logs carry traceId/spanId matching trace spans ▶

What we observed

All three signals are flowing and confirmed. Traces and logs use OTLP gRPC push from the Envoy data plane; metrics use Prometheus pull (controller on port 9092, Envoy proxy on port 9091). Trace-log correlation is confirmed with an exact match: log record traceId=d374625072e87cb0d55ddab5c20eca14 and spanId=565049b8d78c1488 exactly match the corresponding ingress trace span. Metrics confirm OTel pipeline health via envoy_tracing_opentelemetry_spans_sent and envoy_access_logs_open_telemetry_access_log_logs_written counters. The controller emits 21 kgateway_* metrics and the Envoy proxy emits 153+ envoy_* metrics. The controller's structured JSON logs (stdout) are not collected into the OTel pipeline — they remain in Kubernetes log streams only.

What's working well

  • Trace-log correlation is confirmed with an exact traceId/spanId match — operators can drill from an access log record directly to the corresponding trace span in supported backends.
  • The official docs treat all three signals together in a single OTel integration guide, reflecting intentional multi-signal design rather than per-signal afterthoughts.

Opportunities

Route controller logs into the OTel pipeline — The kgateway controller emits structured JSON logs to stdout, but these are not collected into the OTel pipeline. Forwarding controller logs via OTLP (or at minimum via the OTel Collector's filelog receiver) would complete the multi-signal picture for the control plane.
Resolve service.name mismatch for trace-log correlation — While traceId/spanId correlation works, the service.name differs between traces (kgateway-envoy.demo) and logs (demo-gateway.demo) for the same Envoy proxy. Fixing this inconsistency would enable service-identity-based correlation in addition to trace-context-based correlation.
1 Audience & Signal Quality Span names partially useful; Envoy-internal naming leaks into span names and attributes ▶

What we observed

Envoy span names are ingress (the inbound listener span) and router kube_demo_otel-eval-backend_3000 egress (the upstream connection span). The ingress name carries no HTTP method or route context — unlike the OTel HTTP semconv recommendation of {http.request.method} {http.route} (e.g., GET /). The egress span name exposes Envoy's internal cluster naming format (kube_<namespace>_<service>_<port>). Log bodies are always empty ({}) — all access log data (HTTP method, path, status, latency) is absent from OTel log records. Several span attributes carry low-value data: downstream_cluster: "-", zone: "", response_flags: "-". The kgateway controller metrics (kgateway_translator_translations_total, kgateway_controller_reconcile_duration_seconds) are well-named and operationally focused.

What's working well

  • The trace structure (ingress → router ... egress → backend) tells a coherent story of a request's journey through the gateway — interpretable even without deep Envoy knowledge.
  • The kgateway controller's 21 kgateway_* metrics are operationally focused and well-named, covering reconciliation latency, XDS snapshot stats, and resource sync counts — useful for platform engineers.

Opportunities

Improve span naming with request context — The ingress span name could include the HTTP method and route (e.g., GET /api/v1) to match OTel HTTP semconv recommendations. The egress span name could use a human-readable upstream service name instead of the Envoy cluster format kube_demo_otel-eval-backend_3000.
Populate OTel access log body with request data — OTel access log records currently have an empty body ({}) and no log attributes. Adding HTTP method, path, status code, response time, and upstream address to the log body or attributes would make OTel access logs as useful as Envoy's traditional JSON access logs.
1 Stability & Change Management Observability documented in official docs with OTel guide; no explicit telemetry stability contract or changelog sections ▶

What we observed

kgateway has dedicated observability documentation at /docs/envoy/latest/observability/otel-stack/, /docs/envoy/latest/observability/control-plane-metrics/, and /docs/envoy/latest/observability/gateway-metrics/. The docs describe how to configure OTel telemetry but do not establish it as a stable contract. No schema URL is set in native OTLP exports for traces or logs (the schemaUrl: https://opentelemetry.io/schemas/1.18.0 visible in metrics data is added by the OTel Collector's Prometheus receiver, not by kgateway). The v2.2.4 release notes contain no telemetry-specific change sections. There is no explicit stability classification for telemetry fields (stable vs. experimental), and no migration guidance for telemetry changes. The developer architecture docs (devel/architecture/metrics.md) describe internal metrics design but are contributor-facing, not a user-facing stability contract.

What's working well

  • Observability is treated as a first-class feature with dedicated documentation sections covering all three signals — a strong foundation for formalizing stability commitments.
  • OTel is explicitly positioned as the recommended integration path in the official docs, giving operators a clear signal about the project's observability direction.

Opportunities

Add telemetry change sections to release notes — Adding a dedicated "Telemetry changes" section to release notes would help operators who rely on specific metric names, span attribute names, or log formats to plan upgrades and avoid surprises when the telemetry schema evolves.
Establish a telemetry stability contract — Explicitly labeling which telemetry fields are stable vs. experimental (in the docs or as code annotations), and setting schemaUrl in OTLP exports, would give operators confidence to build automation and dashboards on top of kgateway's telemetry without fear of silent breakage.
Key takeaways

● Strengths

  • Multi-signal observability with confirmed trace-log correlation — All three signals flow, and log records carry exact traceId/spanId values matching the corresponding ingress trace spans — enabling drill-down from logs to traces in supported backends.
  • End-to-end W3C Trace Context propagation — Envoy injects traceparent headers into upstream requests, and the backend correctly continues the trace. The full ingress → router egress → backend trace tree is visible in a single trace, confirmed with real telemetry data.
  • Well-documented OTel integration path — The official docs have a dedicated "Set up the OpenTelemetry stack" guide covering all three signals with concrete configuration examples, positioning OTel as the recommended observability path.

● Priority improvements

  • Migrate to current HTTP semantic conventions — Track Envoy's upstream semconv migration to replace deprecated http.method, http.url, http.status_code (as string) with current http.request.method, url.full, http.response.status_code (as integer).
  • Fix service.name inconsistency between traces and logs — The Envoy proxy emits kgateway-envoy.demo in traces but demo-gateway.demo in logs for the same pod. Aligning these to the same configured value would enable service-identity-based cross-signal correlation.
  • Populate OTel access log body with request data — Envoy OTel ALS log records have an empty body ({}) and no attributes. Including HTTP method, path, status code, and latency would make OTel access logs a first-class signal for request-level analysis.
  • Add OTLP metrics export path — Neither the kgateway controller nor the Envoy proxy currently supports OTLP metrics push. Adding an OTLP exporter to the controller would complete the integration surface and eliminate the Prometheus scrape dependency for metrics.

● Notable observations

  • spawnUpstreamSpan is not the default — Without spawnUpstreamSpan: true in ListenerPolicy, only the ingress span is generated. The default should arguably be true for a proxy that exists specifically to forward requests upstream.
  • Envoy metrics port is 9091, not the default 9901 — kgateway's deployer overrides Envoy's default admin port (9901) with a custom stats endpoint at port 9091. This is automatically annotated on the pod and works correctly, but may surprise operators familiar with standard Envoy deployments.
  • Metrics schemaUrl is collector-derived, not project-native — The schemaUrl: https://opentelemetry.io/schemas/1.18.0 visible in metrics data is added by the OTel Collector's Prometheus receiver, not by kgateway. This gives an appearance of schema awareness that is collector-provided, not project-native.
