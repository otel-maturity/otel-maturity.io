🔍 Dimension Details

01 Integration Surface Level 2 — OTel-Native ▶

What We Observed

Tekton supports OTLP export for both traces and metrics. Traces flow via otlptracehttp to a configurable endpoint, controlled by the config-tracing ConfigMap (enabled: "true", endpoint: "http://host:port/v1/traces"). Metrics default to Prometheus scrape on :9090 with optional OTLP export (grpc or http/protobuf) via config-observability. Both paths are documented in official Tekton docs and ConfigMap _example sections. Logs remain stdout-only with no OTLP path.

A notable documentation gap exists: the config-observability ConfigMap's _example section shows tracing-protocol and tracing-endpoint keys that are not used by the actual OTLP trace exporter — only config-tracing controls trace export. This can mislead operators.

What's Working Well

OTLP is supported and configurable for both traces and metrics without code changes — via ConfigMap patching alone.
Configuration is Kubernetes-native (ConfigMap-based), which fits naturally into GitOps and Helm-based deployment workflows.
Official documentation at tekton.dev/docs/pipelines/metrics/ covers both Prometheus and OTLP options with clear examples.

Opportunities

Adding OTLP log export (even as opt-in) would complete the three-signal story and enable log-trace correlation.
Correcting or removing the misleading tracing-protocol / tracing-endpoint keys from the config-observability _example section would prevent operator confusion.
Supporting OTEL_EXPORTER_OTLP_ENDPOINT as an environment variable would align with standard OTel SDK configuration patterns and improve portability.
02 Semantic Conventions Level 1 — OTel-Aligned ▶

What We Observed

Span-level attributes use custom, non-namespaced keys: taskrun, namespace, and pipelinerun. These appear only on root spans — child spans (prepare, createPod, reconcile, etc.) carry no attributes at all. The schema URL is set to https://opentelemetry.io/schemas/1.12.0 (early 2023) while the current stable version is v1.27.0. Instrumentation scope versions report "unknown".

Metric names follow a custom tekton_pipelines_controller_* convention with no OTel semconv alignment. All Tekton metrics have unit: null. The _milliseconds suffix on taskruns_pod_latency_milliseconds is inconsistent with the _seconds suffix used on other duration metrics.

What's Working Well

The schema URL is present on resource spans — a positive signal that Tekton is aware of the semconv versioning model.
No deprecated OTel attributes are used in Tekton's own spans (the deprecated HTTP attributes observed come from the separate otel-eval-backend service).
Infrastructure metrics (kn_workqueue_*, go_*) were recently migrated from OpenCensus and now align better with standard naming conventions.

Opportunities

Replacing namespace with k8s.namespace.name and introducing namespaced keys like tekton.taskrun.name and tekton.pipelinerun.name would align with OTel conventions while preserving Tekton-specific semantics.
Adding attributes to child spans (e.g., k8s.namespace.name on createPod) would improve filtering and querying in observability backends.
Updating the semconv import from v1.12.0 to v1.27.0 would ensure any future semconv-aligned attributes start from a current baseline.
Setting metric units (s for seconds, ms for milliseconds, {taskrun} for counts) would make metrics self-documenting and improve compatibility with OTel-aware backends.
Setting the instrumentation scope version to the Tekton release version (e.g., v1.12.0) would improve telemetry provenance.

Concrete Examples

  • Current: taskrun = "call-eval-backend-run-26" → Suggested: tekton.taskrun.name = "call-eval-backend-run-26"
  • Current: namespace = "default" → Suggested: k8s.namespace.name = "default"
  • Current: pipelinerun = "eval-pipeline-run-7" → Suggested: tekton.pipelinerun.name = "eval-pipeline-run-7"
  • Metric unit gap: tekton_pipelines_controller_taskrun_duration_seconds has unit: null → should be unit: "s"
03 Resource Attributes & Configuration Level 1 — OTel-Aligned ▶

What We Observed

Tekton's createTracerProvider function sets only service.name natively (values: "taskrun-reconciler" and "pipelinerun-reconciler"). The Go OTel SDK does not auto-populate telemetry.sdk.language, telemetry.sdk.name, telemetry.sdk.version, service.version, or service.instance.id when using a custom resource constructor. No OTEL_* environment variables are set on controller pods — all configuration is via ConfigMap.

Identity is inconsistent across signals: traces use service.name: "taskrun-reconciler" while metrics use service.name: "tekton-pipelines-controller" (assigned by the Prometheus receiver from the job label). This prevents cross-signal correlation by service.name.

What's Working Well

service.name is set natively on traces — the minimum required for signal routing in most backends.
The ConfigMap-based configuration approach is idiomatic for Kubernetes operators and integrates naturally with GitOps workflows.
After collector enrichment via the k8sattributes processor, rich Kubernetes metadata is added: k8s.pod.name, k8s.namespace.name, k8s.deployment.name, k8s.node.name, and more.

Opportunities

Using resource.Default() instead of resource.NewWithAttributes() would automatically populate telemetry.sdk.language, telemetry.sdk.name, and telemetry.sdk.version from the Go OTel SDK.
Adding service.version from the release label (v1.12.0) would allow operators to filter telemetry by Tekton version — especially useful during upgrades.
Aligning service.name between traces and metrics (e.g., both using "tekton-pipelines-controller") would enable cross-signal correlation without collector-side join logic.
Supporting OTEL_SERVICE_NAME and OTEL_RESOURCE_ATTRIBUTES environment variables would allow operators to override resource attributes using standard OTel patterns.

Concrete Examples

  • Traces: service.name = "taskrun-reconciler"
  • Metrics: service.name = "tekton-pipelines-controller" (set by Prometheus receiver)
  • Missing natively: service.version, service.instance.id, telemetry.sdk.language, telemetry.sdk.name, telemetry.sdk.version
04 Trace Modeling & Context Propagation Level 2 — OTel-Native ▶

What We Observed

Tekton produces rich, hierarchical span trees that accurately represent the CI/CD lifecycle. A typical TaskRun trace includes: TaskRun:Reconciler → TaskRun:ReconcileKind → prepare, reconcile (→ createPod), finishReconcileUpdateEmitEvents, and durationAndCountMetrics. Multiple reconcile loops for the same TaskRun appear within a single trace, which is correct behavior.

W3C Trace Context propagation is implemented via propagation.TraceContext{}. Inbound context propagation is supported via the tekton.dev/taskrunSpanContext annotation on TaskRun objects. Outbound propagation stores the W3C traceparent in pod annotations, enabling downstream workloads to continue the trace. All spans use kind=INTERNAL.

What's Working Well

The tekton.dev/taskrunSpanContext annotation mechanism is an elegant solution for propagating trace context across the Kubernetes controller boundary — acknowledging the architectural reality that CI/CD work happens in separate pods.
The trace hierarchy is coherent and follows the actual code execution path, making traces immediately useful for debugging slow or failing pipelines.
Multiple reconcile loops for the same TaskRun are correctly grouped under a single root span, providing a complete per-TaskRun trace.

Opportunities

Considering SpanKind.SERVER for root reconciler spans (TaskRun:Reconciler, PipelineRun:Reconciler) would better reflect that these spans represent the entry point of handling a Kubernetes event.
Adding documentation and tooling guidance for workloads that want to consume the tekton.dev/taskrunSpanContext pod annotation would make end-to-end tracing more accessible.
Supporting the standard traceparent annotation (not just tekton.dev/taskrunSpanContext) on TaskRun objects would allow external systems to inject trace context using the standard W3C header name.

Concrete Examples — Span Tree

TaskRun:Reconciler (root, INTERNAL)          ← taskrun=..., namespace=...
  └── TaskRun:ReconcileKind (INTERNAL)
        ├── prepare (INTERNAL)
        │     └── updateTaskRunWithDefaultWorkspaces
        ├── reconcile (INTERNAL)
        │     └── createPod (INTERNAL)
        ├── finishReconcileUpdateEmitEvents (INTERNAL)
        │     └── updateLabelsAndAnnotations
        └── durationAndCountMetrics (INTERNAL)
05 Multi-Signal Observability Level 1 — OTel-Aligned ▶

What We Observed

Two signals are present (traces and metrics) but use different collection paths and have inconsistent resource attributes that prevent correlation. Traces use OTLP push while metrics use Prometheus pull — the split means the two signals cannot easily be correlated by the collector. Logs are entirely absent from the OTel signal.

Cross-signal correlation is not possible: traces use service.name: "taskrun-reconciler" while metrics use service.name: "tekton-pipelines-controller". Controller logs contain knative.dev/traceid which is a UUID-format internal Knative ID — completely separate from OTel hex-format trace IDs.

What's Working Well

Both traces and metrics are available and provide genuine value independently — traces for debugging individual runs, metrics for aggregate CI/CD performance monitoring.
The tekton.dev/taskrunSpanContext pod annotation provides a theoretical path for pod-level workloads to correlate with the controller trace, enabling end-to-end tracing in OTel-aware workloads.

Opportunities

Aligning service.name between traces and metrics would be the highest-impact single change for enabling cross-signal correlation.
Adding the OTel traceId and spanId to structured log output (even via stdout) would enable log-trace correlation without requiring full OTLP log export.
Introducing OTLP log export (opt-in) would complete the three-signal story and enable full observability in OTel-native backends like Grafana or Honeycomb.
Unifying the collection model (e.g., supporting OTLP push for metrics as the default) would simplify collector configuration and enable cross-signal enrichment at the collector level.
06 Audience & Signal Quality Level 2 — OTel-Native ▶

What We Observed

The majority of span names reflect meaningful CI/CD lifecycle operations: TaskRun:Reconciler, createPod, prepare, stopSidecars, resolvePipelineState, runNextSchedulableTask, createTaskRun. A minority of spans are implementation-specific: finishReconcileUpdateEmitEvents, updateLabelsAndAnnotations, durationAndCountMetrics, updatePipelineRunStatusFromInformer.

Tekton-specific metrics are directly actionable for CI/CD operators: duration histograms with status labels enable SLA monitoring; running count gauges enable capacity monitoring; pod latency metrics enable scheduling performance monitoring. Tracing is disabled by default; metrics are on by default.

What's Working Well

The Tekton-specific metrics (tekton_pipelines_controller_taskrun_duration_seconds, _pipelinerun_duration_seconds, _taskruns_pod_latency_milliseconds) directly measure CI/CD performance and are immediately useful for SLA monitoring and capacity planning.
The metrics.taskrun.level and metrics.pipelinerun.level settings allow operators to control metric cardinality — important for high-volume Tekton deployments.
The trace volume is appropriate: ~8-10 spans per reconcile loop, ~30-40 spans per complete TaskRun lifecycle — manageable without overwhelming backends.

Opportunities

Renaming or removing the durationAndCountMetrics span would reduce noise — this is an internal implementation detail, not a meaningful CI/CD operation.
Enabling tracing by default (or providing a simpler one-step enablement path) would make traces more accessible to operators who don't know to configure config-tracing.
Adding attributes to child spans (e.g., pod name on createPod, task name on runNextSchedulableTask) would significantly improve the diagnostic value of individual spans.

Concrete Examples — Key Metrics

  • tekton_pipelines_controller_taskrun_duration_seconds — histogram with status, namespace, task labels
  • tekton_pipelines_controller_pipelinerun_duration_seconds — histogram with status, namespace, pipeline labels
  • tekton_pipelines_controller_taskruns_pod_latency_milliseconds — scheduling latency histogram
  • tekton_pipelines_controller_running_taskruns — gauge for capacity monitoring
  • tekton_pipelines_controller_running_pipelineruns_waiting_on_pipeline_resolution — resolution bottleneck gauge
07 Stability & Change Management Level 2 — OTel-Native ▶

What We Observed

Tekton has dedicated observability documentation at tekton.dev/docs/pipelines/metrics/ and a dedicated migration guide tekton.dev/docs/pipelines/metrics-migration-otel/ for the OpenCensus → OpenTelemetry transition. All core Tekton metrics are explicitly labeled "experimental" in the official documentation. Schema URL is present on traces (v1.12.0, outdated) and metrics (v1.18.0, set by the collector). No formal telemetry stability contract exists.

What's Working Well

The OpenCensus → OTel migration guide (PR #9043) is exemplary: it provides explicit breaking change tables with old-name → new-name mappings, impact levels (HIGH/MEDIUM/LOW), and action items for operators.
The "experimental" label on all metrics is honest about stability — operators know not to rely on metric names without checking for breaking changes.
Configuration options are documented in both ConfigMap _example sections and official docs, giving operators two discovery paths.

Opportunities

Introducing a formal telemetry stability contract (e.g., "metrics marked stable will not be renamed without a deprecation period") would give operators confidence to build dashboards and alerts on Tekton metrics.
Updating the trace schema URL from v1.12.0 to v1.27.0 would ensure the schema URL reflects current conventions.
Correcting the documentation confusion between config-observability (which shows tracing-protocol/tracing-endpoint in its example but does NOT control trace export) and config-tracing (which does) would prevent a significant operational footgun.
Removing the Jaeger-specific default endpoint (http://jaeger-collector.jaeger.svc.cluster.local:4318/v1/traces) from config-tracing in favor of a generic example would reduce backend coupling in documentation.

💡 Key Takeaways

Strengths

  • Rich trace trees for CI/CD lifecycle — hierarchical traces accurately represent TaskRun/PipelineRun reconciliation and are immediately useful for debugging.
  • Thoughtful context propagation design — the tekton.dev/taskrunSpanContext annotation elegantly bridges the controller-to-pod boundary for end-to-end tracing.
  • Exemplary migration documentation — the OpenCensus → OTel migration guide with explicit breaking change tables sets a high bar for telemetry change communication.
  • Actionable CI/CD metrics — duration histograms, running counts, and pod latency metrics directly measure pipeline performance and are immediately useful for SLA monitoring.
  • Flexible cardinality control — metrics.taskrun.level and metrics.pipelinerun.level settings allow operators to manage metric volume in high-traffic deployments.

Priority Improvements

  • Align service.name across signals — the single highest-impact change for enabling cross-signal correlation. Traces use "taskrun-reconciler"; metrics use "tekton-pipelines-controller".
  • Add service.version and telemetry.sdk.* — use resource.Default() to auto-populate SDK attributes and add version from the release label.
  • Fix the config-observability documentation footgun — the misleading tracing-protocol/tracing-endpoint keys in config-observability cause operators to misconfigure tracing.
  • Namespace span and metric attribute keys — replace taskrun → tekton.taskrun.name, namespace → k8s.namespace.name to align with OTel conventions.
  • Set metric units — all Tekton metrics have unit: null; adding "s", "ms", and "{taskrun}" units is a low-effort, high-value improvement.

Notable Observations

  • knative.dev/traceid is NOT an OTel trace ID — it's a UUID-format internal Knative reconciler identifier. Operators expecting log-trace correlation will be misled.
  • The config-tracing vs config-observability split is an operational footgun — configuring tracing via config-observability (which appears to support it) silently does nothing for OTLP traces.
  • Semconv v1.12.0 is ~3 years outdated — the tracing code imports go.opentelemetry.io/otel/semconv/v1.12.0; current stable is v1.27.0.
  • The pod annotation is a well-designed interface — tekton.dev/taskrunSpanContext: {"traceparent":"00-...-01"} stores the W3C trace context, enabling OTel-aware workloads to continue the trace.
  • Instrumentation scope version is "unknown" — setting it to the Tekton release version would improve telemetry provenance.

Methodology

  • OTel Collector v0.150.1 with file exporter in a kind cluster (otel-eval-tekton)
  • 17 JSONL trace batches; 160 JSONL metric batches; 0 OTLP log records
  • 30 TaskRuns (incl. 5 with external trace context) + 8 PipelineRuns generated
  • Source reviewed: pkg/tracing/tracing.go, pkg/reconciler/taskrun/tracing.go
  • Semconv checked against OTel semconv v1.27.0 (current stable)
